## 安全な設計原則

### 1.1 オブジェクトとサブジェクト

情報セキュリティでは、**サブジェクト**（誰か）と**オブジェクト**（何か）の関係がアクセス制御の基本です。

- **サブジェクト**：リソースにアクセスする人やプログラム（例：ユーザー、プロセス）。
- **オブジェクト**：データやリソース（例：ファイル、プリンター、データベース）。

このモデルは「**誰が何をどうできるか**」を決める基盤で、DAC、MAC、RBACなどのアクセス制御に使われます。

#### サブジェクトとオブジェクトの関係
アクセス制御システムは、サブがオブとどう関わるかを管理します。例えば：
- ユーザーが文書を読む
- プロセスがログに書き込む
- プログラムが別のプロセスを起動

| 用語       | 説明                                     |
|------------|------------------------------------------|
| サブject   | アクティブにアクセスを要求するもの         |
| オブject   | アクセスされるデータやリソース             |
| アクセス   | 実行される操作（読む、書く、実行など）   |

🧠 **CISSPポイント**：すべてのアクセス制御は、サブとオブを使って権限や制限を定義します。

#### 動的な役割
あるものは状況によってサブにもオブにもなります。
- 例：
  - プロセスA（サブ）→ プロセスB（オブ）
  - プロセスB（今度はサブ）→ プロセスC（オブ）

この柔軟性は**マルチスレッドシステム**や**プロセス間通信**で重要です。

#### 推移的信頼
**推移的信頼**は、AがBを信頼し、BがCを信頼すると、AがCを間接的に信頼する状態です。
- これが管理ミスで**意図しないアクセス**や**セキュリティの穴**になることがあります。

🚨 **試験の注意**：クラウドや連合システムでは推移的信頼が危険。信頼の連鎖を常に確認！

#### 1.1 Access Rules

| モデル | 説明 |
|---|---|
| **任意アクセス制御 (DAC)** | オブジェクトの所有者がアクセス権を設定 |
| **強制アクセス制御 (MAC)** | ラベル（例：極秘）に基づくアクセス制御 |
| **役割ベースアクセス制御 (RBAC)** | ユーザーの役割に応じたアクセス制御 |

🛡️ **CISSP試験のヒント**：どのモデルがサブ/オブをどう使うか、ペアの評価方法を理解してください。

#### まとめ
- サブ-オブモデルは「誰が何をどうするか」を定義。
- アクセス制御、システム設計、セキュリティモデルの基本。
- この関係を理解すると、リスクポイントや最小権限の適用が簡単になります。

| 概念               | 概要                                       |
|--------------------|--------------------------------------------|
| サブject         | アクティブなアクセス者（ユーザー/プロセス） |
| オブject           | パッシブなターゲット（ファイル/デバイス）   |
| アクセス制御     | サブがどのオブにどんなアクションを起こせるかを制限 |
| 推移的信頼       | 間接的な信頼がシステムに広がる可能性。危険   |
| 動的行動         | 同じエンティティが状況でサブ/オブに変わる |

### 1.2 クローズドシステムとオープンシステム

クローズドとオープンシステムは、**どれだけ内部が見えるか、制御できるか**を表します。これにより、相互運用性、セキュリティの確認、信頼度が変わります。

#### クローズドシステム
- **非公開で制限された**システム。
- 内部の仕組みは公開されません。
- 通常、**1つの組織やベンダー**が開発。
- アップデートや統合には**ベンダー依存**。
- **ベンダーロックイン**や透明性の低さが課題。

**メリット**：
- 集中管理
- 一貫性とサポート

**デメリット**：
- 脆弱性の監査が難しい
- サードパーティとの連携が制限

🔒 **例**：Windowsのソースコードはクローズド。Microsoftだけが変更可能。

#### オープンシステム
- **公開インターフェースや標準**で透明性が高い。
- **相互運用性**や**コミュニティ協力**を促進。
- ソースコードやドキュメントが**公開**されることが多い。
- **オープン標準**に対応し、統合しやすい。

**メリット**：
- 監査やテストが簡単
- コミュニティによる改善
- ベンダー依存が少ない

**デメリット**：
- 実装品質にバラつき
- メンテナンスやサポートに手間

🌍 **例**：UbuntuのようなLinuxディストリビューションはオープンシステム。

#### APIと相互運用性
**API（アプリケーションプログラミングインターフェース）**は、オープン/クローズドシステム間で**相互運用性**を可能にします。

- 安全で制御されたアクセスを提供。
- オープンシステムは外部開発者向けにAPIを公開。
- クローズドシステムは**制限的または独自のAPI**を提供。

🧩 **相互運用性**は以下に重要：
- マルチベンダー環境
- ベンダーロックインの回避
- ハイブリッドアーキテクチャ（クラウド＋オンプレミス）

#### オープンソース vs クローズドソース

| カテゴリ       | オープンソース                         | クローズドソース                     |
|----------------|---------------------------------------|-------------------------------------|
| アクセス       | 公開で改変可能なコード               | ベンダーが管理する独自コード         |
| 透明性         | 高（誰でもコードを検査可能）         | 低（隠蔽によるセキュリティリスク）   |
| コミュニティ   | 協力的な開発                         | 内部開発                           |
| カスタマイズ   | 高自由度                           | 制限あり                           |
| サポート       | コミュニティやベンダー（バラつき）   | 公式ベンダーサポート               |
| ライセンス     | GPL、MIT、Apacheなど               | ベンダー独自のEULA                 |

💡 **CISSPポイント**：オープンソースは必ずしも安全ではない。**コード品質**、**パッチ管理**、**コミュニティサポート**に依存。

#### まとめ
- クローズドは一貫性と制御を提供するが、透明性が低い。
- オープンは相互運用性と検査を促進するが、注意が必要。
- APIは両者を橋渡しし、安全な統合を可能に。
- 試験では**リスク**、**信頼**、**統合の複雑さ**をシステムタイプで評価する問題が出るかも。

🛡️ **CISSP試験のヒント**：オープンソース＝デフォルトで安全と混同しない。各モデルの適切な用途を理解。

### 1.3 安全なデフォルト設定

**安全なデフォルト設定**は、システムやアプリを**最初から最も安全な状態**で設定し、**カスタマイズや展開前に攻撃面を最小化**することです。

多くの侵害は、デフォルト設定が**安全でない**、変更されない、または過度に寛容なために起こります。

#### なぜ重要か
システムが以下のような状態で出荷されると：
- 開放されたポート
- デフォルト認証情報
- 無制限のアクセス
…攻撃者にとって**簡単な標的**になります。

安全なデフォルトは**最小権限の原則**を体現—ユーザー、プロセス、サービスは**必要最低限のアクセス**のみ持つべきです。

🛡️ **CISSPポイント**：安全なデフォルトはユーザーが強化する前から保護を適用。積極的な防御策です。

#### 安全なデフォルトの原則
- **不要なサービスを無効化**：サービスが少ない＝攻撃ベクトルが少ない。
- **制限的なアクセス制御**：デフォルトで拒否、信頼された行動のみ明示的に許可。
- **ゲストアカウントやデフォルトパスワードを無効化**：攻撃でよく悪用される。
- **ログと監査をデフォルトで有効**：追跡可能性を確保。
- **安全なインストールパス**：一時的またはユーザー書き込み可能なディレクトリでの実行を回避。
- **最小限の機能セット**：必要な機能のみ有効化。

#### 制限的なデフォルト
**制限的デフォルト**は、**すべて拒否し、明示的に許可されたものだけを通す**姿勢です。

- 例：ファイアウォールはルールで許可されていないトラフィックをすべてドロップ。
- **意図的で制御された設定**を促進。
- **「デフォルトで拒否」**モデルに従う。

| 設定スタイル       | 説明                                         | セキュリティ姿勢     |
|--------------------|----------------------------------------------|---------------------|
| デフォルトでオープン | サービスやアクセスが広く有効                 | ❌ 高リスク          |
| 制限的デフォルト   | 明示的に許可されるまでアクセス不可           | ✅ 安全で監査可能   |

🚨 **試験のリマインダー**：システムはユーザーが手動で安全に設定するのに頼るべきではない。**デフォルト**はすでに強化されているべき。

#### 安全なデフォルトの一般的な違反
- **SSHやRDPをインターネットに公開**
- **admin/admin**や**root/toor**のデフォルト認証情報
- ウェブサーバーの**ディレクトリリスティング有効**
- データベースの**空またはデフォルトパスワード**

✅ **修正**：設定管理とコンプライアンススキャンで安全な設定を強制。

#### まとめ
安全なデフォルトは以下でリスクを軽減：
- 危険な機能を無効化
- 最小権限を強制
- デフォルトでアクセスを拒否
- 意図的で安全な設定を促進

| 概念                 | 概要                                                                 |
|----------------------|----------------------------------------------------------------------|
| 安全なデフォルト     | 可能な限り安全な状態で事前設定                                       |
| 制限的デフォルト     | デフォルトで全て拒否、必要なアクセスのみ明示的に許可               |
| 一般的な違反         | 不要なサービス有効、デフォルトパスワード、開放ポート                 |
| リスク軽減           | 安全なデフォルトは展開後のユーザーによる強化への依存を軽減           |

🔑 **CISSP試験のヒント**：**セキュリティはオプトアウトでなくオプトイン**であるべき。常に**安全なデフォルト**を優先。

### 1.4 安全に失敗する

**安全に失敗する**とは、システムやコンポーネントがエラーや障害時に**安全な状態**（開放的でない状態）にデフォルトで移行することです。

この原則は、**異常な状況**（停電、アプリクラッシュ、アクセス拒否など）でも**セキュリティが保たれる**ことを保証します。

#### 安全に失敗するの主な概念
- **認証が検証できない場合**、アクセスを拒否。
- **システムがクラッシュや停電した場合**、機密データが公開されない。
- **入力検証でエラーが起きた場合**、入力を**拒否**（受け入れない）。

🛡️ **CISSP試験のポイント**：システムはセキュリティ制御を回避したり、機密データを公開するような失敗をしてはいけない。

#### フェイルオープン vs フェイルクローズド

| 失敗の挙動       | 説明                                         | セキュリティへの影響 |
|------------------|----------------------------------------------|---------------------|
| **フェイルオープン** | 障害時にアクセスや機能を許可               | ❌ 高リスク          |
| **フェイルクローズド** | 障害時にアクセスを拒否し、動作を停止       | ✅ 安全な選択        |

**フェイルクローズド**が安全な推奨挙動です。

**例**：
- ファイアウォールはルールが読み込めない場合、**フェイルクローズド**でトラフィックをブロック。
- アクセス制御は認証失敗時にエントリを拒否、バイパスを許可しない。

#### フェイルセーフ vs フェイルセキュア

これらの用語は似ていますが、微妙な違いがあります：

| 用語          | コンテキスト           | 障害時の挙動                                 |
|---------------|------------------------|----------------------------------------------|
| フェイルセーフ | **安全第一**           | 人の安全を優先（例：火災時にドアをアンロック） |
| フェイルセキュア | **セキュリティ第一**   | セキュリティを維持（例：火災時でもドアをロック） |

⚠️ **CISSP試験の注意**：フェイルセーフ＝物理的安全、フェイルセキュア＝データ/セキュリティ保護。

#### ストップエラーとBSoD
- **ストップエラー**（例：Windowsの**ブルースクリーン・オブ・デス（BSoD）**）は、重大な障害時にシステムを停止。
- 混乱を招くが、**さらなる損害や破損を防ぐ**意図的動作。

🧠 **セキュリティ視点**：
- クラッシュは**機密性と完全性を保持**しない限り安全ではない。
- システムは障害をログに記録し、監査証跡を保持し、**メモリ内のデータ公開**を防ぐ必要がある。

#### まとめ
安全に失敗することは、問題が起きたときに**安全かつセキュア**に動作を停止し、データ漏洩や不正アクセスを防ぎます。

| 概念                 | 概要                                                                |
|----------------------|--------------------------------------------------------------------|
| フェイルクローズド   | 障害時にアクセス拒否またはシステム停止（安全なデフォルト）           |
| フェイルオープン     | 障害時にアクセス許可（危険、安全上必須でない限り避ける）             |
| フェイルセーフ       | 物理的安全を優先（例：緊急時に出口ドアをアンロック）                 |
| フェイルセキュア     | システム/データセキュリティを優先（例：停電時もロック維持）         |
| BSoD/ストップエラー  | さらなるエラーを防ぐ強制停止—完全性を保つのに役立つ可能性           |

✅ **CISSP試験のヒント**：制御を設計する際、「これが失敗したら、システムを保護するか、公開するか？」を自問。

### 1.5 シンプルかつ小さく保つ（KISS）

**KISS（Keep It Simple and Small）**原則は、シンプルな設計を重視し、**セキュリティ**、**信頼性**、**メンテナンス性**を高めます。セキュリティアーキテクチャでは、**複雑なシステムは保護、監査、理解が難しく**、設定ミスや見逃した脆弱性のリスクが増えます。

🧠 **CISSPポイント**：複雑さはセキュリティの敵。シンプルな設計は**デフォルトで安全**で、評価やテストが容易。

#### 原則の概要
- 軍事や工学から：「シンプルなシステムは失敗が少ない」。
- 最小限の機能と単純なロジックを推奨。
- 小さなコードベースやシステムは**攻撃面が少ない**。
- エラー、設定ミス、隠れたバックドアの可能性を低減。

#### セキュリティにおけるKISSの利点

| 利点                 | 説明                                                                 |
|----------------------|----------------------------------------------------------------------|
| 攻撃ベクトルの減少   | 複雑さが少ない＝潜在的な攻撃が少ない                                 |
| 監査の容易さ         | シンプルなシステムは理解、テスト、検証が簡単                         |
| 迅速なパッチ適用     | コードが少ないとバグの発見と修正が早い                             |
| メンテナンス性の向上 | 変更時のエラー導入リスクが低い                                     |
| 可用性の向上         | 部品が少ないと失敗の可能性が低い                                   |

#### 関連する設計原則

##### DRY（Don’t Repeat Yourself）
- コードやロジックの重複を避ける。
- 変更は1か所だけで済む—エラーや不整合を減少。

##### YAGNI（You Aren’t Gonna Need It）
- 絶対に必要な機能だけ実装。
- 使われない機能は**デッドコード**を生み、攻撃面を増やす。

##### セキュリティのミニマリズム
- 公開するモジュール、サービス、ポートを最小限に。
- **最小限のデフォルトインストール**を使い、不要な機能をオフに。

🛡️ **試験のヒント**：機能が少ないシステムがKISSに従い、リスクを最小化するシナリオが出題されるかも。

#### 実際の例
- 多数の不要なサービスを持つフル機能メールサーバーではなく、送信専用メールリレーの軽量MTAを展開。
- Windowsの使わない機能やサービスを「念のため」稼働させず、無効化。

#### まとめ
**KISS原則**は**簡素化**でセキュリティを支援。複雑さを減らすことで：
- 脆弱性を制限
- 監査を簡素化
- システムの予測可能な動作を保証

| 概念         | 概要                                                          |
|--------------|--------------------------------------------------------------|
| KISS         | シンプルで最小限のシステム設計                                |
| DRY          | 冗長性を避け、更新を簡素化しリスクを軽減                    |
| YAGNI        | 現在必要ない機能は作らない                                    |
| ミニマリズム | サービスや機能を減らし、攻撃面を小さく                        |

🔍 **CISSPリマインダー**：KISSはアプリコードだけでなく、**システムアーキテクチャ、ポリシー、セキュリティ設定**にも適用。

### 1.6 ゼロトラスト

**ゼロトラスト**は、以下に基づく現代のセキュリティアーキテクチャとマインドセットです：  
> **「決して信頼せず、常に検証する。」**

**内部も外部も、ユーザー、デバイス、システムを自動的に信頼しない**と仮定。すべてのアクセス要求は、アイデンティティ、コンテキスト、リスクに基づいて**継続的に検証**される必要があります。

🛡️ **CISSP試験のポイント**：ゼロトラストは単なる製品ではなく、**アイデンティティ検証、アクセス制御、セグメンテーション、監視**を含む戦略。

#### ゼロトラストの核心原則

1. **明示的に検証**  
   - 利用可能なすべてのデータ（ユーザーID、デバイスの状態、場所など）で常に認証と認可。

2. **最小権限のアクセスを使用**  
   - ユーザーやデバイスのアクセスを役割に必要な最小限に制限（最小権限の原則）。

3. **侵害を前提**  
   - 攻撃者がすでにネットワーク内にいる前提でシステムを設計。

#### ゼロトラストの主要コンポーネント

| コンポーネント           | 説明                                                                 |
|--------------------------|--------------------------------------------------------------------|
| アイデンティティ管理（IAM） | 強力な認証（MFA）、継続的なセッション検証                         |
| ネットワークセグメンテーション | 脅威を封じ込めるためにネットワークを小さなセグメントに分割       |
| エンドポイントセキュリティ | デバイスのコンプライアンス確保（例：最新パッチ、EDRエージェント） |
| 監視と分析               | リアルタイムで異常や潜在的侵害を検出                             |

#### マイクロセグメンテーション
ネットワークを**細かいゾーン**に分割し、**ワークロードやプロセスレベル**でアクセス制御を適用。

- 横移動を制限
- 侵害を隔離
- **次世代ファイアウォール（NGFW）**、**ソフトウェア定義ネットワーク（SDN）**、**ホストベースのファイアウォール**で実施

🧱 **例**：ゲストVLANのプリンターが財務サーバーにアクセスできない。

#### リアルタイム監視
ゼロトラストは**継続的な監視**を重視し、認証、ネットワーク動作、データアクセスを追跡。

- **UEBA（ユーザー行動分析）**や**SIEM**を活用。
- ダメージ前に脅威を検出・対応。
- インシデント対応ワークフローと統合。

📌 **CISSPノート**：ゼロトラストは「何も信頼しない」ではなく、**「すべてを検証する」**ことを意味。

#### エアギャップとファラデーケージ
**物理レベル**でのゼロトラストをサポートするため：
- **エアギャップ**：他のネットワークから完全に切断（例：ICS/SCADA）。
- **ファラデーケージ**：無線データ漏洩や盗聴を防ぐ電磁シールド。

🔒 これらの方法は**秘密チャネル**を防ぎ、物理的隔離を強制。

#### まとめ
ゼロトラストは、**強力な認証、細かい認可、継続的な監視**を強制し、複雑な環境でのリスクを軽減する**戦略モデル**。

| 概念                   | 概要                                                                 |
|------------------------|----------------------------------------------------------------------|
| ゼロトラスト           | 内部でも信頼せず、常に検証                                           |
| マイクロセグメンテーション | 小さな隔離ゾーンで横移動を防止                                     |
| リアルタイム監視       | 行動、アイデンティティ、セッションコンテキストで脅威を検出         |
| エアギャップ/ファラデーケージ | 物理的および電磁的隔離を強制                                     |
| 最小権限の原則         | ゼロトラストの核心—絶対に必要なアクセスのみ                       |

🔐 **CISSPヒント**：試験ではゼロトラストを**包括的なアーキテクチャ**として理解し、技術的制御だけではないことを認識。

### 1.7 信頼するが検証する

**信頼するが検証する**は、最初に信頼しても**継続的な検証**が必要とする古典的なセキュリティ概念です。

冷戦時代の軍備管理交渉で使われたロシアの諺に由来し、サイバーセキュリティで広く採用されています。**慎重なマインドセット**を反映：信頼されたユーザー、システム、プロセスも**継続的なチェック**を受け、期待通りに動作するかを確認。

🧠 **CISSPポイント**：従来の境界ベースのセキュリティとゼロトラストの橋渡しで、**監視**、**監査**、**継続的な保証**の重要性を強調。

#### 核心概念
- **信頼**：アイデンティティ、役割、過去の行動に基づくアクセス許可。
- **検証**：信頼が有効であるかを常に監視・再評価。

**内部脅威**、**認証情報の侵害**、**ポリシーの逸脱**を検出するために重要。

#### セキュリティ設計の例

| シナリオ                           | 信頼                     | 検証メカニズム                       |
|------------------------------------|--------------------------|-------------------------------------|
| VPNアクセスの従業員               | 役割ベースのアクセス許可 | セッションログ、MFAチャレンジ       |
| クラウド管理者アカウント           | 管理者権限の割り当て     | 活動監視、監査証跡                 |
| 署名済みアプリ                   | 信頼された発行者         | 実行時動作監視、ハッシュチェック     |
| 内部システム間通信               | ホワイトリストIPやトークン | 相互TLS、異常検知                   |

#### 「信頼するが検証する」の実装メカニズム
- **監査ログ**：誰がいつ何にアクセスしたかを記録。
- **異常検知**：ベースラインからの逸脱を監視。
- **多要素認証（MFA）**：信頼されたユーザーも再認証。
- **ジャストインタイム（JIT）アクセス**：時間制限付き権限付与。
- **行動分析（UEBA）**：疑わしいユーザー行動を検出。

🚨 **CISSP試験のヒント**：信頼するが検証するは**多層防御**の鍵。ゼロトラストを補完し、置き換えるものではない。

#### 信頼するが検証する vs. ゼロトラスト

| 原則                 | 焦点                                 | デフォルトの姿勢             |
|----------------------|--------------------------------------|-----------------------------|
| 信頼するが検証する   | 継続的に信頼を検証                   | 最初に信頼、その後検証       |
| ゼロトラスト         | デフォルトで誰も信頼しない           | 信頼前に検証                 |

📌 「信頼するが検証する」は**初期の信頼**を前提に継続検証、**ゼロトラスト**は初期の前提を排除。

#### まとめ
**信頼するが検証する**原則は以下で**セキュリティ保証**を維持：
- 継続的な監視
- 行動とアクセスの検証
- 役割と権限の再評価

| 主要な側面           | 概要                                                        |
|----------------------|------------------------------------------------------------|
| 原則                 | 信頼は永遠に仮定せず、獲得・再検証する必要がある           |
| 使用ケース           | 特権ユーザーやクリティカルシステムの継続検証               |
| ツールと技術         | ログ、監視、異常検知、MFA、JITアクセス                     |
| CISSP関連性          | 最小権限と説明責任の原則を強化                             |

### 1.8 プライバシーバイデザイン（PbD）

**プライバシーバイデザイン（PbD）**は、システムやビジネスプロセスの設計とアーキテクチャに**プライバシー保護を最初から組み込む**積極的なセキュリティフレームワークです。

元オンタリオ州情報・プライバシーコミッショナーのアン・カブキアン博士が開発し、現在は**EU GDPR**や多くの**国際プライバシーフレームワーク**でグローバルに認識されています。

🛡️ **CISSP試験のヒント**：PbDは**プライバシーが設計の核心**であることを保証—後付けではない。反応的コンプライアンスから積極的設計への移行。

#### 核心概念
- プライバシーを**尊重・保護**するシステムを構築。
- プライバシー制御を「後から追加」しない。
- **データ最小化**、**ユーザー同意**、**透明性**に焦点。

#### PbDの7つの基本原則

| 原則                                   | 説明                                                                 |
|----------------------------------------|----------------------------------------------------------------------|
| 反応的でなく積極的、修復的でなく予防的 | プライバシー侵害を事前に予測・防止                                   |
| デフォルトでプライバシー               | 個人データはユーザー操作なしで自動保護                               |
| 設計に組み込まれたプライバシー         | プライバシーは後付け機能でなくシステムの核心                         |
| 完全な機能—ポジティブサム             | プライバシーとビジネス目標を両立（ゼロサムでない）                   |
| エンドツーエンドのセキュリティ         | データ収集から削除まで保護                                         |
| 可視性と透明性                         | ユーザーやステークホルダーに操作の透明性を提供                       |
| ユーザープライバシーの尊重             | 強力なプライバシーデフォルト、通知、ユーザー中心の選択肢             |

🧠 **試験のポイント**：PbDは**技術的手段だけでなく**、**ポリシー、ガバナンス、文化**も含む。

#### グローバルプライバシースタンダード（GPS）
**GPS**は、**プライバシーの期待と保護の一貫性**を促進する国際協定。

- PbD原則に基づく。
- 異なる管轄区域で**プライバシー慣行を調和**。
- 一貫したプライバシー保護の下で国境を越えたデータ転送をサポート。

📌 GPSは**プライバシー調整フレームワーク**—特に多国籍企業にとって重要。

#### システムアーキテクチャでのPbD
以下で適用：
- **必要最低限のデータ収集**（データ最小化）。
- 保存/送信データの**匿名化**または**暗号化**。
- 明確な**ユーザー同意フロー**と**プライバシー通知**。
- **ログ**、**監査**、**プライバシー影響評価（PIA）**の組み込み。

#### PbD実装の例

| シナリオ                   | PbD実装例                                        |
|----------------------------|--------------------------------------------------|
| ウェブアプリ               | オプトインのデータ収集、同意なしのトラッキング禁止 |
| 医療システム               | 研究用に匿名化されたレコード                     |
| モバイルアプリ             | デバイスリソースへのパーミッションベースアクセス   |
| クラウドストレージ         | エンドツーエンド暗号化とユーザー制御のデータアクセス |

#### まとめ
**プライバシーバイデザイン**は、プライバシーが**システムに組み込まれ**、後付けでないことを保証。

| 核心概念             | 概要                                                                  |
|----------------------|----------------------------------------------------------------------|
| 哲学                 | 実装後でなく設計時にプライバシーを組み込む                           |
| 核心価値             | 透明性、制御、ユーザー同意、全ライフサイクルセキュリティ             |
| 標準                 | 7つの基本原則とグローバルプライバシースタンダード                     |
| CISSP関連性          | SDLC、リスク管理、プライバシー規制、システム設計に適用                 |

### 1.9 セキュアアクセスサービスエッジ（SASE）

**セキュアアクセスサービスエッジ（SASE）**は、**ネットワークとセキュリティ機能**を統合し、クラウド配信サービスとして提供する現代のセキュリティアーキテクチャ。2019年にガートナーが提唱し、従来の境界ベースモデルから**分散型、クラウドネイティブ、モバイル環境**への移行に対応。

🧠 **CISSPポイント**：SASEは**アイデンティティ駆動型、クラウド対応のセキュリティ**のニーズを反映し、場所やデバイスに関係なく安全なアクセスを可能に。

#### 核心特性
SASEは以下のサービスを統合：
- **SD-WAN**：インテリジェントなルーティング
- **ゼロトラストネットワークアクセス（ZTNA）**：アイデンティティベースのアクセス制御
- **クラウドアクセスセキュリティブローカー（CASB）**：クラウドアプリの監視
- **サービスとしてのファイアウォール（FWaaS）**：クラウドでの境界防御
- **セキュアウェブゲートウェイ（SWG）**：ユーザーのインターネットアクセス保護

これらは**クラウドからサービスとして提供**され、どこからでも一貫したセキュリティとアクセスポリシーを適用。

#### クラウドネイティブとアイデンティティ中心
SASEは以下を重視：
- **クラウドネイティブ**：需要に応じて弾力的にスケール、オンプレミスハードウェア依存を排除。
- **アイデンティティ中心**：**ユーザーID**、**デバイスの状態**、**コンテキスト**、**リスクレベル**に基づくアクセス—IPアドレスや物理的場所ではない。

📌 **ゼロトラスト**原則に沿い、**リモートワーク**、**BYOD**、**ハイブリッドクラウド**環境をサポート。

| 従来のネットワーク境界 | SASEアーキテクチャ                 |
|-----------------------|------------------------------------|
| 集中型、静的         | 分散型、動的                       |
| IP/場所ベースの制御   | アイデンティティ/コンテキストベース |
| アプライアンスベース   | クラウド配信のセキュリティサービス   |

#### ZTNA、エッジコンピューティング、監視

##### ゼロトラストネットワークアクセス（ZTNA）
- VPNを置き換え、**細かいアイデンティティベースのアクセス**を特定アプリに提供。
- 最小権限を強制し、**デフォルトで信頼しない**。
- ユーザー、デバイス、コンテキストを継続検証。

##### エッジコンピューティング
- 処理をユーザー/デバイスに近づける（エッジ）ことで遅延を減らし、パフォーマンスを向上。
- SASEノードはグローバルエッジに配置され、アクセスとポリシー適用を最適化。

##### 継続的な監視と分析
- **リアルタイム監視**、**脅威検知**、**ポリシー強制**を含む。
- **SIEM**、**SOAR**、**UEBA**と統合し、動的なリスクベースのアクセス決定。

#### SASEの利点

| 利点                       | 説明                                                                 |
|----------------------------|----------------------------------------------------------------------|
| スケーラビリティ           | グローバルな成長企業に対応する弾力的なクラウドインフラ                 |
| 統合セキュリティ           | 単一ベンダー下で複数の機能を統合                                     |
| 一貫したポリシー強制       | ユーザー場所やデバイスに関係なく同じセキュリティルール               |
| パフォーマンス向上         | 低遅延ルーティングとエッジ最適化                                     |
| コスト効率                 | オンプレミスの複数セキュリティアプライアンスの必要性を削減             |

#### まとめ
**SASE**は、ネットワークとセキュリティを**クラウド配信、アイデンティティ駆動型モデル**に統合する大きな変化。

| 機能                 | 概要                                                                 |
|----------------------|----------------------------------------------------------------------|
| アーキテクチャ       | クラウドネイティブ、分散型、スケーラブル                             |
| セキュリティモデル   | ゼロトラスト原則に基づく                                             |
| 主要コンポーネント   | ZTNA、CASB、FWaaS、SWG、SD-WAN                                       |
| 利点                 | 統合制御、複雑さ軽減、リモートセキュリティ強化                       |
| CISSP関連性          | 現代の企業課題（リモートワーク、クラウド優先）に適合                   |

## 2. CIAを確保する技術

**機密性（Confidentiality）**、**完全性（Integrity）**、**可用性（Availability）**（CIA）は情報セキュリティの基本目標。セキュリティアーキテクトは、ソフトウェアやプロセスの動作とリソースとの相互作用を制御する**システム設計技術**を適用し、CIAを確保します。これらの原則は、**隔離**、**境界**、**分離**、**アクセス制御**モデル、**信頼/保証**フレームワークで強化されます。

### 2.1 隔離

**隔離**（サンドボックスとも呼ばれる）は、プロセスの**メモリやシステムリソースへのアクセスを制限**する手法。

- プロセスは**許可されたメモリ位置**やリソースにのみアクセス可能。
- 境界を超える試みは拒否、ログ記録、場合によっては終了。

🛡️ 隔離は**最小権限の原則（PoLP）**の適用—各プロセスは必要なものだけにアクセス。

📌 **実装例**：
- OSレベルのプロセス分離
- **Sandboxie**などのセキュリティサービス
- 仮想化ソリューション（例：VMware、VirtualBox）

🧠 **CISSPヒント**：隔離はOSレベルで強制され、**データ漏洩**や**不正アクセス**を防ぐ。

### 2.2 境界

**境界**は、プロセスやサブルの**システム内の権限の限界**を定義。

- **範囲**は、サブが読み書きや実行できる**最大のアクセス範囲**。
- OSはプロセスに論理的または物理的メモリセグメントを割り当てるために境界を使用。

| 種類                | 説明                                         |
|---------------------|----------------------------------------------|
| 論理的境界           | 共有メモリ空間でのOSメカニズムによる制限     |
| 物理的境界           | 物理的に分離されたメモリセグメント           |

📌 **CISSPポイント**：境界は**プロセスの権限レベル**（例：ユーザーモード vs カーネルモード）に基づいて設定。

### 2.3 分離

**分離**は、プロセスの動作が**他のプロセスに影響を与えない**環境で実行を保証。

- **隔離と境界**を強制することで実現。
- **偶発的干渉**と**意図的悪用**を防ぐ。

分離の利点：
- 1つのプロセスが失敗してもシステム全体に影響なし。
- **OS環境**、**カーネル**、他のアプリを保護。

📌 分離は**プロセスセキュリティの基盤**で、**フェイルソフトシステム**や**障害封じ込め**をサポート。

| メカニズム         | 目的                                    |
|----------------------|----------------------------------------|
| ハードウェア分離     | 物理メモリオーバーラップを防止       |
| OSレベル分離         | プロセス間のクロス干渉を防止           |
| 仮想化   | 全体のOS環境をサンドボックス化         |

### 2.4 アクセス制御

**アクセス制御**は、**許可されたサブだけが特定のオブに、承認された方法で**アクセス可能にします。

- アクセスは**ルールやポリシー**に基づいて管理。
- **アクセス制御システム**で評価される。

#### 一般的なモデル
| モデル | 説明 |
|---|---|
| **任意アクセス制御 (DAC)** | 所有者が権限を設定 |
| **強制アクセス制御 (MAC)** | 機密ラベル（例：極秘）に基づく |
| **役割ベースアクセス制御 (RBAC)** | 割り当てられた役割に基づく |
| **属性ベースアクセス制御 (ABAC)** | ユーザー、リソース、環境の属性に基づく |

🧠 **CISSP試験のヒント**：アクセス制御は、**誰が何をいつどのようにできるか**を制限し、CIAトライアドを強制。

### 2.5 信頼と保証

**信頼**：システムがセキュリティポリシーを確実に強制するという信念。

**保証**：セキュリティ対策が正しく実装され、意図通りに機能するという**確信**。

主要ポイント：
- 信頼は**セキュリティメカニズム**の実装で構築。
- 保証は**テスト**、**検証**、**認証**（例：コモンクライテリア、EALレベル）で測定。

🛠️ **保証の維持**：
- システム変更後（例：アップデート、侵害）の**再検証**。
- **パッチ管理**、**設定管理**、**変更制御**のベストプラクティスに従う。

| 用語       | 説明                                                         |
|------------|-------------------------------------------------------------|
| 信頼       | セキュリティ機能の存在                                       |
| 保証       | その機能の有効性と信頼性に対する自信                         |

📌 **CISSPポイント**：保証はシステム変更後も継続的に見直し・維持が必要。

### まとめ

| 技術             | 目的                                                         |
|------------------|-------------------------------------------------------------|
| 隔離             | プロセスのメモリ/リソース空間を制限                         |
| 境界             | サブまたはプロセスの最大許可アクセスを定義                   |
| 分離             | あるプロセスの他のプロセスへの干渉を防止                   |
| アクセス制御     | 誰が何にいつどうアクセスできるかを強制                     |
| 信頼と保証       | セキュリティ機能の信頼性と強さを評価                       |

これらの技術は**CIAを共同で強化**し、安全で安定した強制可能なシステムの設計に不可欠。

## 3. セキュリティモデルの基本概念を理解する

### 3.1 トークン、機能、ラベル

セキュアシステムでは、アクセス制御の決定は、サブとオブのセキュリティ属性を記述するメタデータに基づきます。これを表現・強制する3つの主要メカニズム：

#### トークン
**セキュリティトークン**は、リソースの**アクセス制御属性**を持つ独立したエンティティ（例：ファイル、データ構造）。

- サブまたはオブに関連し、権限を示す。
- リソースアクセス前に使用。
- 異なるコンポーネントやセッション間で再利用・共有可能。

📌 **例**：セッションログイン時に発行される一時的な認証情報（ユーザーID、役割、権限を含む）。

🧠 **CISSPポイント**：トークンはアクセス管理を簡素化するが、改ざんやリプレイ攻撃から保護が必要。

#### 機能
**機能リスト**は、**サブ中心**の構造で、**サブがさまざまなオブに対して実行可能な操作**を定義。

- 各サブは複数のオブに対する権限リストを保持。
- サブのアクセス決定が速く、オブの検索不要。
- トークンより柔軟性が低い（リストはサブに紐づく）。

| 概念         | 焦点           | 保管場所             |
|--------------|----------------|---------------------|
| トークン     | オブベース       | オブに関連           |
| 機能         | サブベース       | サブに関連           |

🧠 **CISSPヒント**：機能システムでは、オブのアクセス取り消しは**各サブのリスト更新**が必要で管理が大変。

#### ラベル
**セキュリティラベル**は、オブの分類や感度を示す**永続的で埋め込まれたタグ**。

- **必須アクセス制御（MAC）**で使用。
- ラベル例：「機密」、「極秘」など。
- 通常のユーザーによる変更不可。

📌 **例**：MACシステムで「極秘」ラベルのファイルは、適切なクリアランスがないとアクセス不可。

🛡️ セキュリティラベルは**多レベルセキュリティ**を強制し、**不正開示**を防ぐ。

#### まとめ
| メカニズム   | 説明                                            | 使用シナリオ                     |
|--------------|-------------------------------------------------|
| トークン     | リソースにリンクされたセキュリティメタデータ     | セッショントークン、Kerberosチケット |
| 機能       | サブが持つオブの権限リスト                   | 分散システム、マイクロカーネル     |
| ラベル       | オブに埋め込まれた分類レベル                   | 政府MACシステム、SELinux           |

これらのメカニズムを理解すると、**ポリシーの適用方法**、**誰が何にアクセス可能か**、**権限の強制方法**が明確に。

### 3.2 トラステッドコンピューティングベース（TCB）

**Trusted Computing Base (TCB)**は、セキュアシステムの基盤で、システムのセキュリティポリシーを強制する**すべてのハードウェア、ソフトウェア、制御**を含む。

#### 主要特性
- **小さく、検証可能**：TCBは**最小限**で、**分析・検証・信頼**できるようにする。
- **必須コンポーネント**：TCBのみが**アクセス制御を強制**、認証情報を検証、保護されたシステムリソースを管理。
- **隔離**：外部コンポーネントからの改ざんを防ぐ**保護境界**内で動作。

🛖 **CISSPヒント**：システムの信頼性は**TCBの強さ**に依存。弱いTCBは全体のセキュリティモデルを損なう。

#### TCBの責任
- **認証**と**認可**の強制
- **メモリ**、**ファイル**、**I/O**の保護
- **すべてのアクセス要求**の仲介
- **セキュアブート**、**プロセス隔離**、**信頼パス**の提供

| TCBの役割         | 説明                                   |
|--------------------|----------------------------------------|
| ポリシー強制     | すべてのアクセスと操作がポリシーに準拠 |
| アクセス検証     | オブアクセス前にサブの認証情報を確認   |
| リソース管理     | 保護されたメモリ、ファイル、デバイス管理 |

#### セキュリティ境界
- **セキュリティ境界**は、TCBを囲む**仮想の境界**。
- 外部（非公開）コンポーネントとの安全でない通信を防ぐ。
- TCBと安全にやり取りするには**信頼パス**が必要。

📌 **信頼パス**：例：WindowsのCtrl+Alt+Del画面はセキュアなログイン用の信頼パス。

#### リファレンスモニター概念
**リファレンスモニター**は、**すべてのアクセス要求**が許可前にチェックされる抽象的な概念。

- 以下の特性：
  - **改ざん防止**
  - **常に呼び出される**
  - **検証可能なサイズ**

✅ **セキュリティカニセル**に実装され、リファレンスモニターの決定を強制するTCBのソフトウェア/ハード部分。

#### まとめ
| 概念               | 説明                                                                |
|--------------------|--------------------------------------------------------------------|
| TCB                | システムの信頼できる核心、セキュリティポリシーを強制             |
| セキュリティ境界   | TCBをシステムの他の部分から隔離                                     |
| 信頼パス           TCBと安全に通信する方法 |
| リファレンスモニター | サブ-オブアクセスのすべてをチェックするコンポーネント |
| セキュリティカニセル | リファレンスモニターの実装                                         |

🧠 **CISSP試験のポイント**：TCB、セキュリティ境界、リファレンスモニター、セキュリティカニセルの相互関係とアクセス制御の強制方法を理解。

### 3.3 状態マシンモデル

**状態マシンモデル**は、システムが安全な状態から始まり、**安全な状態のみを遷移する**場合にセキュアとみなす基本的なセキュリティモデル。他の多くのセキュリティモデル（例：ベル・ラパデュラ、ビバ）の基礎。

#### 主要概念
- **状態**：ある時点でのシステムのスナップショット。
- **安全な状態**：**すべてのセキュリティルールとポリシー**が強制されている状態。
- **状態遷移**：入力がシステムの状態を変更する。
- システムは**すべての遷移が安全な状態に導く**場合のみセキュア。

📌 **式**：  
`次の状態 = F(現在の状態, 入力)`  
`出力 = F(現在の状態, 入力)`

🧠 **CISSPヒント**：遷移がセキュリティポリシー違反を許すと、モデルは崩れ、不安全に。

#### 安全状態マシンの特性
- **すべてのアクセス要求が処理前に検証**。
- 遷移中（例：プログラム実行、ログイン/ログアウト）でも**一貫したセキュリティ強制**。
- **安全な状態で起動**し、実行中もセキュリティを維持。

| 概念                 | 説明                                     |
|----------------------|------------------------------------------|
| 安全な状態           | すべてのアクセス制御とポリシーが強制     |
| 遷移               | ユーザー入力やシステムイベントで状態が変化 |
| 有限状態マシン       システムの動作をモデル化する構造 |

#### 他のモデルとの関連性
- **ベル・ラパデュラモデル**は、**機密性**を保証するために状態マシンを使用。
- **ビバモデル**は、**完全性**を強制に拡張。
- 時間とともに進化するシステム（**非決定論**や**情報フロー**など）は状態マシンを基盤に。

🛖 **セキュリティ保証**：
- すべての状態と遷移を評価することで、システムが**不安全な設定に入らない**ことを保証。

#### まとめ
| 要素                 | 目的                                    |
|----------------------|----------------------------------------|
| 状態                 | システムの状態の瞬間                    |
| 安全な状態             | すべてのポリシーを強制                  |
| 遷移                 | 入力/イベントで状態を変更              |
| 安全状態マシン       | すべての状態で安全な操作を保証        |

🔍 **試験のヒント**：状態マシンモデルはシステム動作の**数学的表現**で、ほとんどの正式なセキュリティモデルの**基盤**。

### 3.4 情報フローモデル

**情報フローモデル**は、システム内の**情報の移動を制御**し、**不正、不安全、不意の流れ**を防ぐセキュリティモデル。**状態マシンモデル**の原則を基盤に、**多層セキュリティシステム**に拡張。

#### 主な目標
- 情報が**許可された方向**にのみ流れる。
- **異なるセキュリティレベル**間のデータ漏洩を防止。
- **秘密チャネネル**や未定義の通信経路をブロック。

📌 **例**：機密システムで、「機密」ユーザーが「極秘」データを受け取ってはいけない—間接的にも。

#### 核心特性
- **すべてのデータ移動パス**を評価。
- **誰が何にアクセスできるか**だけでなく、**データがどう動くか**も制御。
- **多レベルセキュリティ（MLS）**システムに特に有用で、サブとオブに分類レベルがある。

#### 情報の流れの種類
| フロー Type       | 説明                                                                 |
|-------------------|----------------------------------------------------------------------|
| 明示的フロー      | 直接的に定義されたデータ移動（例：コピー/ペースト）                   |
| 暗黙のフロー   | 条件ロジックやプログラム状態による間接的影響                         |
| 秘密チャンネル    | 意図されない方法でデータが流れる隠れ経路                     |

#### 時間ベースの視点
- 情報の流れモデルは**時間を通じてデータを追跡**し、オブの状態を異なる時点で比較：
  - 低分類オブが**高分類でから汚染されていない**ことを確認。
  - **歴史的な完全性と機密性**を検証。

🛖 **CISSP試験のヒント**：このモデルが**データ漏洩を制御・防止**する方法と、アクセス制御ベースのモデルとの違いを理解。

#### 多レベルセキュリティ（MLS）
以下のような階層的分類システムで情報量制御が必須：
- トップシークレット
- シークレット
- 機密
- 非公開

これらのシステムでは、ユーザーやファイルだけでなく、アプリやメモリ位置でも**不正な流れをブロック**。

#### 秘密チャンネル保護
情報フロー模型の主要目標は**秘密チャンネルの特定と排除**：
- **タイミングチャンネル**：CPUサイクルや応答時間で情報漏洩
- **ストレージチャンネル**：共有ストレージ（例：未使用ファイルスペース）でメッセージ送信

| 機能                 | 説明                                                                 |
|----------------------|-----------------------------------------------------------------------------|
| フロー制御         | 情報の伝達方法を制限                                                 |
| 時間状態リンク       | オブジェクトのバージョンの履歴整合性を確保                           |
| 秘密チャンネル軽減   | 意図しない通信パス（タイミング、ストレージなど）をブロック             |

#### まとめ
- **データの動き**に焦点を当て、アクセス権だけでなく。
- 高保証システムで**情報漏洩を防止**。
- **直接および間接**の情報フローを評価。
- **軍事**、**政府**、**クラウド**環境で重要。

🧠 **記憶術**：  
"**フロー = 制御 + クラス + チャンネル**"  
制御を強化、分類を尊重、チャンネルを閉じる。

### 3.5 非干渉モデル

**非干渉モデル**は、**高セキュリティレベルの行動が低レベルで観察や推測**されないようにする正式なセキュリティモデル。**情報漏洩を防止**し、セキュリティドメイン間の**厳格な隔離**を維持。

#### 核心概念
- 非侵入は、**不許可のサブ**がシステムの動作を観察して**機密活動**を知るのを防ぐ。
- **高レベル行動**（例：機密操作）は、低レベルユーザーに見えたり、知覚される影響を与えてはいけない。
- **行動的隔離**に焦点、アクセス権ではない。

📌 **CISSPポイント**：アクセスをブロックするだけでなく、**低レベルユーザーが高レベル行動の存在すら検知できない**ようにする。

#### 例シナリオ
**多層セキュリティシステム**で：
- **トップシークレット**ユーザーがファイルを削除やプログラムを実行。
- **非クラシファイド**ユーザーは、システム性能低下、ファイル数の変化、ログ更新で何かが起こったことを**観察**してはいけない。

🚖 **目標**：**クリアランス境界を越える観測可能な副作用**や干渉がないこと。

#### 主要機能
| 機能                   | 説明                                                                 |
|-------------------------|--------------------------------------------------------------------|
| 行動的隔離             | 低レベルでの高レベル活動の影響を防止                         |
| 秘密チャンネルブロック | 意図しない通信パス（例：タイミングやストレージ漏洩）を排除   |
| 推論保護           | 間接的な証拠から機密情報の推測を阻止                 |

#### 適用領域
- **オペレーティングシステム**：プロセス間漏洩を防止
- **仮想化**：クラウド環境でテナントを隔離 |
- **軍システム**：絶対的な分類完全性を強制 |

🔑 **実世界例**：
仮想化環境で、VM1のテナントがVM2の機密活動による性能変化を検知できれば、非干渉に違反。

#### 情報フローモデルとの違い
| モデル                 | 焦点                              | セキュリティ目標                     |
|------------------------|------------------------------------|-------------------------------------|
| 情報フロー            | データの流れの方向と方向性         | 境界間のデータ漏洩を防止           |
| 非干渉                 | 行動の副作用と可視性             | 推論と秘密信号の防止               |

#### まとめ
- **非干渉＝レベル間の影響なし**。
| - Ensures **low-level users remain unaware** to high-level operations.
| - Critical in **センサリー級** or **cloud tenant isolation** requirements.
| - Supports **軍事-grade** and **cloud** separation needs.

🖌 **CISSP試験のヒント**：このモデルが従来のアクセス制御を超え、**推論や間接開示**をブロックすることを理解。

### 3.6 構成理論

**構成理論**は、複数のシステムが結合したときにセキュリティプロパティがどのように保持されるかを記述します。**個々のセキュアコンポーネント**が大きなシステムに統合されたときに**セキュアに動作する**が重要です。

#### 核心目標
異なるシステムの**入力と出力**が互いに影響し、結果として生じる複合システムで**セキュリティ違反**が起きないかを理解。

#### 3つの主要な構成理論
| 理論             | 説明                                                                                     | 例使用ケース                           |
|------------------|------------------------------------|
| **カスケード**       | 1つのシステムの出力が次のシステムの入力に。                                         | セキュアなプリンターが監視システムにログを渡す |
| **フィードバック**        | 2つのシステムが入力と出力を循環的または双方向に交換。                     | アイデンティティプロバイダー（IdP）がユーザーを認証、その後ユーザーの行動がIdPの動作を変更 |
| **フックアップ**       | 複数のシステムが単一のシステムまたは外部に送る。                             | IoTデバイスが中央サーバーにテレメトリーデータを送信 |

#### 適用コンテキスト
| **マルチシステム環境** |
| **セキュリティアーキテクチャ設計** |
| **相互接続されたエンタープライズネットワーク** |
| **連合システムとクラウドサービスチェーン** |

🔗 **セキュリティ懸念**：
各システムが独自のポリシーを守る場合でも、情報や信頼の流れが結合すると**予期しないパス**、**推論リスク**、**秘密チャンネル**が導入される可能性。

#### 他のモデルとの比較
| モデル                     | 焦点                              | 適用                                  |
|--------------------------|------------------------------------|--------------------------------------|
| 構成理論            | セキュアシステムの統合             | 全体システムのセキュリティ評価       |
| Structured Machine Model | 状態遷移によるセキュリティ         | セキュアブート、遷移検証ストレージ |
| 非干渉モデル            | 高/低レベルの隔離                 | ドメイン間の影響防止                 |

#### システム構成を評価しないと**システム的脆弱性**が生じる可能性があります。

🖗 **CISSP試験のヒント**：**3つの構成種類の違い**と、**なぜ個別にセキュリティされたシステムが統合時にリスクを伴うか**を理解。

### 3.7 テイクグラントモデル

**テイクグラントモデル**は、**サブとオブの間で権利がどう移転する**かを分析するグラフベースのアクセス制御モデル。**アクセス権の分布と流れ**を追跡し、**権利の漏洩や拡散**を評価。

#### 核心コンポーネント
- **Subjects**：アクションを実行するエンティティ（例：ユーザー、プロセス）。
- **サブjects**：アクセスされるパッシブエブ（例：ファイル、デバイス）。
- **権利**：アクセス許可（例：読む、書く、実行、所有）。
- **グラフ構造**：エンティティをノード、権利をノード間のエッジとして表現。

#### 4つの主要ルール

| ルール         | 説明                           |
|----------------|-------------------------------|
| **テイク**     | サブは他のサブ/オブからでも権利を取れる |
| **Grant**     | サブは、持っている権限を他のサブ/オブに付与 |
| **Create**     | サブは、新たなオブやサブを作成し、権限を割り当てる |
| **Remove**     | サブは、持っているまたは割り当てた権限を取り消す |

📌 **「テイク」**は他の許可を引き取り、**「グラント」**は持っているものを与えると考えて。

#### グラフ表現
- **ノード**：サブまたはオブジェクト
- **エッジ**：アクセス権（例：読む、書く、テイク、グラント）

このグラフを**探索**し、サブが直接割り当てられていない場合でも特定のアクセス権を得られるかを判断。

#### 実際の影響
- **アクセス制御の進化**を示す。
- OSのグループ権限の継承分析に有効。
- **特権回避**や**権限漏洩**の可能性を特定。

🔒 **セキュリティ分析の例**：
低権限ユーザーが連鎖的なテイク/グラント操作で高レベルアクセスを得られるかを判断。

#### 他のモデルとの比較
| モデル                  | 焦点                              | メモ                                  |
|---------------------|------------------------------------|--------------------------------------|
| テイクグラント       | 権限の移転と拡散                 | 権利の拡散や漏洩のモデリングに最適   |
| ベルラパデュラ | 機密性の保護                     | 分類に基づく不正読/書を防止       |
| システム               | 完全性の保全                     | 不適切なデータ書き込みによる汚染を防止 |
| アクセス制御マトリクス | 静的アクセス権限                   | サブ-オブ権をマップ、流れはモデルしない |

#### システム構成を識別するのに**特権エスケープパス**と**右漏洩**。

| **テイクアウト**：グラフを用いた動的アクセス制御モデルで、**テイク**、**グラント**、**クリエイト**、**リムーブ**のルールに基づく。

### 3.8 アクセス制御マトリクス

**アクセス制御マトリックス**は、サブシステムで**アクセス権を定義・強制**する基本モデル。サブ、オブ、権限の関係を**表形式**で表現し、**静的許可の割り当て**を理解するのに有用（例：DAC、RBAC、MAC）。

#### マトリックスの構造
- **科目**：ユーザー、プロセス、またはアクセスを要求するエンティティ。
- **オブジェクト**：保護が必要なファイル、デバイス、データベース。
- **許可**：許可された操作（例：読む、書く、実行、削除）。

マトリックスの各セルは、**サブが特定のオブに対して実行可能なアクション**を定義。

|           | File A      | Printer    | Network Share |
|-----------|-------------|-----------------|----------------|
| Alice     | 読む、書く | アクセスなし   | 読む             |
| Bob       | 読む         | 印刷         | 読む、書く         |
| Charlie   | アクセスなし | 印刷、印刷 | 完全制御         |

📌 **システムポイント**：アクセス制御マトリックスは概念的フレームワークで、スケーラビリティの懸念から文字通り表として実装されない場合も。

#### 2つの派生構造
実際の実装を最適化するために、システムはマトリックスから 2 つの構造を派生します。

1. **アクセス制御リスト（ACL）**
   - オブジェクト中心
   - 特定のオブジェクトに対するすべてのサブジェクトとその権限を一覧表示します
   - 例：
     ```プレーンテキスト
     ファイルA: アリス - 読み取り、書き込み; ボブ - 読み取り
     「」

2. **機能リスト**
   - 主題中心
   - すべてのオブジェクトとサブジェクトが持つ権限を一覧表示します
   - 例：
     ```プレーンテキスト
     Bob: ファイルA - 読み取り、プリンター - 印刷、共有 - 読み取り/書き込み
     「」

| 比較 | アクセス制御リスト (ACL) | 機能リスト |
|------------------|------------------------------------------|------------------------------------------|
| フォーカス | オブジェクト | 主題 |
| ユースケース | ファイルシステムで一般的 (例: NTFS) | 直接使用されることはほとんどなく、学術的な価値のみ |
| 管理の容易さ | リソース所有者にとって簡単 | オブジェクトごとに取り消しが困難 |

#### 利点
- シンプルで直感的な構造
- アクセス関係の**完全な可視性**を提供します
- 幅広いアクセス モデル (DAC、RBAC、MAC) をサポート

#### 制限事項
- **スケーラビリティ**: ユーザーやリソースが多数あるシステムでは扱いにくくなります。
- **管理オーバーヘッド**: マトリックスを手動で管理するのは非効率的です。
- **失効の複雑さ** (機能付き): 個々の主体を追跡する必要があります。

#### セキュリティモデルでの使用
- 権限がユーザーによって割り当てられる **DAC** システムでは一般的です。
- **RBAC** の基礎であり、役割は抽象化された主体として機能します。
- オブジェクト ラベルとサブジェクト クリアランスによってアクセスを制御する **MAC** を適用するように拡張できます。

🛡️ **CISSP試験のヒント**: **ACL（オブジェクト中心）**と**機能リスト（サブジェクト中心）**の違いを理解しておきましょう。設問では、これらの正しい使用方法と管理への影響を判断する能力が問われることがよくあります。

#### まとめ
- アクセス制御マトリックスは、サブジェクト、オブジェクト、権限の関係をモデル化します。
- ACL (オブジェクトごと) と機能リスト (サブジェクトごと) に分けることができます。
- さまざまなアクセス制御モデルをサポートするために広く使用されています。
- **誰が何にアクセスできるか**を評価し、**最小権限の原則**を保証します。

| コンセプト | 説明 |
|---------------------|-----------------------------------------------------|
| アクセス制御マトリックス | ユーザーとオブジェクト間のアクセス権の表形式ビュー |
| ACL | オブジェクトベースの権限リスト |
| 機能リスト | サブジェクトベースの権限リスト |

### ベル・ラパドゥーラモデル
**ベル・ラパデュラ（BLP）モデル**は、1970年代に米国国防総省のために開発された、最も初期かつ最も影響力のある**機密性重視のセキュリティモデル**の一つです。**マルチレベルセキュリティ（MLS）**システムにおけるアクセス制御を強化するために設計された**ステートマシンモデル**です。

#### コアフォーカス
- **主な目標**: 情報の**機密性**を保護する
- **整合性** や **可用性** については**考慮されていません**
- **セキュリティクリアランス**と**オブジェクト分類**に基づいて厳格なアクセス制限を実施します

#### 主要なプロパティ
Bell–LaPadula モデルは、異なるセキュリティ レベル間での情報漏洩を防ぐために、**3 つの主要なプロパティ** を適用します。

| プロパティ | 説明 |
|----------------------------------|------------------------------------------------------------------------------|
| **シンプル セキュリティ プロパティ** | 「No Read Up」(ss プロパティ): サブジェクトは、より高いセキュリティ レベルのデータを読み取ることができません |
| *** (スター) プロパティ** | 「書き込み禁止」 (* プロパティ): サブジェクトは、より低いセキュリティ レベルにデータを書き込むことはできません |
| **任意のセキュリティ プロパティ** | アクセスはアクセス制御マトリックス (DAC に類似) によって制御されます |

📌 **CISSP インサイト**: これらのルールは、上位レベルから下位レベルへの **データ漏洩** (例: 「極秘」から「機密」へ) を防ぎます。

#### シナリオ例
Alice が「機密」権限を持っている場合:
- ✅「機密」および「非機密」ファイルを読むことができます
- ❌「秘密」や「極秘」が読めない
- ✅「秘密」または「極秘」に書き込むことができます
- ❌「非機密」への書き込みはできません（機密データの漏洩を防ぐため）

#### 信頼できる主体
**信頼できる主体** は *-property から **免除** され、書き込むことができます (例: 機密解除タスク)。
- 例: レベル間でデータを安全に転送する権限を持つシステム管理者

⚠️ **試験のヒント**: 信頼できる主体は慎重に検証する必要があります。これは、実際の侵害でよく悪用されます。

#### マルチレベルセキュリティ（MLS）システム
BLP はクリアランス レベルによるデータの **区分化** をサポートします。
- 「トップシークレット」
- "秘密"
- "機密"
- 「非公開」

MLS は、必要に応じたアクセス制御と **必須アクセス制御 (MAC)** を実施します。

#### ラティスベースのアクセス制御
BLP は **格子モデル** 内で動作します。
- 各オブジェクトとサブジェクトには**セキュリティラベル**が割り当てられます
- ラベルは階層的に並べられています
- サブジェクトのラベルがオブジェクトのラベルを**支配**している場合にのみアクセスが許可されます

#### 図の概要
Bell-LaPadula セキュリティ ルール:
+----------------------+
| シンプルなセキュリティ: |
| リードアップなし (NRU) |
+----------------------+
| スタープロパティ: |
| 減損なし（NWD）|
+----------------------+

#### 強みと弱み
| 強み | 弱み |
|---------------------------------------|------------------------------------------|
| 強力な機密性を適用します | **整合性** または **可用性** には対応していません |
| レベル間のデータ漏洩を防止します | **秘密チャネル** は防止しません |
| 正式に証明され、広く参照されています | 商用整合性の使用ケースには適していません |

#### まとめ
- Bell–LaPadulaは**機密性**を重視した**強制アクセス制御**モデルです。
- **多層セキュリティラベル**に基づく
- 3 つのプロパティを使用します: **シンプル セキュリティ、* (スター)、および任意セキュリティ**
- 多くの政府および防衛システムの基盤

🛡️ **CISSP 試験のヒント**: 質問が **機密性** または **政府データ分類** に関するものである場合、Bell–LaPadula が正しいモデルである可能性があります。

| 用語 | 説明 |
|----------------------------|--------------------------------------------------|
| シンプルなセキュリティ プロパティ | 読み取り禁止 - 対象者がさらに機密データを読むことを防ぎます |
| * (スター) プロパティ | 減損なし - 下位レベルへの漏洩を防止 |
| 任意セキュリティ プロパティ | DAC に類似したマトリックス ベースの制御 |
| 信頼されたサブジェクト | 厳格な信頼ルールの下で * プロパティをバイパスする可能性があります |

### ビバモデル
**Bibaモデル**は、機密性よりも**整合性**を重視した**ステートマシンベースのセキュリティモデル**です。**Bell-LaPadulaモデル**の逆モデルとして開発され、データの**正確性、信頼性、および破損のなさ**を保証するために使用されます。

#### コアフォーカス
- **主な目標**: **データの整合性**を維持する
- **データの不正または信頼できない変更を防止する**ように設計されています
- データの正確性が重要となる**商業**、**金融**、**医療**環境に適用可能

🧠 **CISSP Insight**: データが **改ざんまたは変更** される懸念がある場合は、Biba モデルを適用できます。

#### 主要なプロパティ
| プロパティ | 説明 |
|---------------------------------|------------------------------------------------------------------|
| **シンプルな整合性プロパティ** | 「下位読み取り不可」: サブジェクトは下位整合性レベルのデータを **読み取ることができません** |
| **スター (*) 整合性プロパティ** | 「書き込み不可」: サブジェクトはより高い整合性レベルに **書き込むことができません** |
| **呼び出しプロパティ** | 下位レベルのサブジェクトは、上位レベルのサブジェクトを **呼び出す** (サービスを要求する) ことはできません |

🛡️ これらのルールは、**データ汚染を防止**し、**情報の信頼性を維持**することを目的としています。

#### 仕組み
- **サブジェクト** (例: ユーザーまたはプロセス) は次の操作のみを実行できます。
  - **自身の整合性レベル以上**でデータを**読み取り**ます（下位の読み取りは行いません）
  - **自身のレベルまたは**それ以下のレベル**でデータを書き込む（書き込みなし）

📌 **類推**:
- 整合性レベルは**空気の純度レベル**と考えてください。
- **汚れた空気** (低整合性) が **クリーンルーム** (高整合性) に入るのは望ましくありません。

#### シナリオ例
ボブが「高誠実性」クリアランスを持つ対象者である場合:
- ✅ 彼は「高」または「非常に高」から**読む**ことができます
- ❌ 「中」または「低」から**読むことができません**（下から読むことはできません）
- ✅ 彼は**「高」または「中」に書くことができます**
- ❌ 彼は**「非常に高い」に書くことができない** (書き込み不可)

これにより、**信頼性の低いユーザーやデータ** が **信頼性の高いシステムやレコード** を侵害することがなくなります。

#### ベル・ラパドゥラとの比較
| モデル | フォーカス | 読み取りルール | 書き込みルール |
|------------------|----------------|-------------------|-------------------|
| ベル・ラパドゥーラ | 機密保持 | 事前確認不要 | 書き留め不要 |
| ビバ | 誠実さ | 読み下げなし | 書き込みなし |

#### 強みと弱み
| 長所 | 制限 |
|--------------------------------------------|--------------------------------------------------|
| **高整合性データの破損を防止** | **機密性には対応していません** |
| **商用環境** に適用可能 | **外部からの脅威のみ** を想定 |
| **監査および会計システム** に役立ちます | **アクセス制御の粒度は処理しません** |

#### まとめ
- Bibaモデルは**データの整合性を保証する**ように設計されています
- これはベル・ラパドゥーラの逆である
- ルールには**シンプル整合性**、**スター(*)整合性**、および**呼び出しプロパティ**が含まれます
- **金融**、**ヘルスケア**、**ログ管理**などの分野で役立ちます

🚨 **試験のヒント**: Biba = 整合性。**「読み上げない、書き上げない」** ことを覚えておいてください。これは、データの盗難ではなく、**データ破損** を防ぐのに最適です。

| 用語 | 説明 |
|------------------------------|----------------------------------------------------------------------|
| シンプルな整合性プロパティ | 読み取りダウンなし - 信頼できないソースからの読み取りを防止 |
| スター (*) 整合性プロパティ | 書き込みなし - 整合性の高いデータの汚染を防止 |
| 呼び出しプロパティ | 下位レベルのプロセスが上位レベルのプロセスを呼び出せないようにする |

### クラーク・ウィルソンモデル
**クラーク・ウィルソンモデル**は、**適切なトランザクションの保証**と**職務分離**の実施に重点を置いた**商業整合性モデル**です。データのラベル付けのみに頼るのではなく、ユーザーが**データの変更**を**承認されたプログラム**のみを通じて行えるように制御します。

🧠 **CISSP Insight**: このモデルは、ユーザーがデータを直接操作できないようにすることで **整合性** を強化します。ユーザーは制御されたインターフェースを経由する必要があります。

#### 重要な概念
| コンセプト | 説明 |
|-------------------------------|-------------------------------------------------------------------------------|
| **件名** | システムと対話しようとしているユーザーまたはエンティティ |
| **プログラム/変換手順 (TP)** | 制約されたデータを操作するための唯一の承認された方法 |
| **オブジェクト/制約データ項目 (CDI)** | 有効な状態で保護および維持する必要があるデータ |

これにより、**トリプレット関係**が作成されます。  
**主語 → TP → 目的語**

#### コンポーネント
| コンポーネント | 説明 |
|-----------------------------------|------------------------------------------------------------------------------|
| **制約付きデータ項目 (CDI)** | モデルによって保護されたデータ。TP 経由でアクセスする必要があります |
| **制約のないデータ項目 (UDI)** | まだ検証されていないデータ。外部から入力でき、変換する必要があります |
| **変換プロシージャ (TP)** | CDI を処理および変換し、ビジネス ロジックを適用するプログラム |
| **整合性検証手順 (IVP)** | CDI が有効な状態のままであるかどうかを確認するルーチン |

#### 強制される原則
1. **整形式のトランザクション**:  
   - データは**許可されたプログラム (TP)**を通じてのみ変更できます。
   - TP は整合性違反を防ぐためにテストされ、承認されます。
   
2. **職務の分離**:  
   - 重要なプロセスの異なるステップは、異なるユーザーが実行する必要があります。
   - 誰も完全な制御権を持たないようにすることで、詐欺や不正行為を防止します。

📌 これにより、Clark–Wilson は **銀行、会計、ERP、およびデータベース管理システム** に最適です。

#### 実際の例
金融システムでは:
- **会計担当者** (対象) は顧客残高 (CDI) を直接編集できません
- 取引を記録して検証する**TP**（例：銀行振込アプリ）を使用する必要があります
- **IVP** は毎日すべての残高をチェックし、不正な変更が行われていないことを確認します

#### 比較: クラーク・ウィルソン vs. ビバ
| 特集 | クラーク・ウィルソンモデル | ビバモデル |
|------------------------|------------------------------------------|----------------------------------------|
| 誠実さの焦点 | 商業的誠実さ | 軍隊式の多層的誠実さ |
| ラベル? | いいえ (プログラムとルールを使用) | はい (分類レベルを使用) |
| 強制メカニズム | TP および IVP を通じて | 読み取り/書き込み制限を通じて |
| 現実世界との整合 | 非常に実用的（実際のビジネスで使用） | より理論的 |

#### まとめ
- クラーク・ウィルソン モデルは、次の方法で **データの整合性** を保証します。
  - **変換手順** (有効な操作のみ許可されます)
  - **職務の分離** (誰も完全な制御権を持たない)
  - **制約されたインターフェース** (TP はゲートキーパーとして機能します)
- **商業環境**でより適用可能
- 重要なデータへの**信頼できるプログラムによるアクセス**を強制します

🛡️ **CISSP 試験のヒント**: 3 つの要素を覚えておいてください:  
**Subject → TP → CDI** と 2 つの原則:  
**整形式のトランザクション** + **職務の分離**

### ブリューワーとナッシュモデル
**ブリューワー・ナッシュ・モデル**（別名**「コカ・コーラ・モデル」**、または非公式には**チャイニーズ・ウォール・モデル**（現在は非推奨））は、**利益相反**を防ぐために設計された**動的アクセス制御モデル**です。**ユーザーの過去のアクティビティに基づいてアクセス決定が変更されることを保証します**。

🧠 **CISSP Insight**: このモデルは、複数の競合するクライアントを扱うことで倫理的な問題や情報漏洩が発生する可能性がある法律事務所、コンサルティング会社、金融機関などの組織にとって重要です。

#### コア目標
特に**競争的なビジネス分野**において、主体（ユーザー）が**利益相反**を引き起こす可能性のあるデータにアクセスするのを防ぐため。

> **例**: コカ・コーラと協力するコンサルタントは、たとえ承認されていても、コカ・コーラの機密ファイルにすでにアクセスしている場合は、ペプシのデータにアクセスできません。

#### 重要な概念
| 用語 | 説明 |
|---------------------------|----------------------------------------------------------------------------|
| **利益相反クラス** | 競合または対立関係にあると見なされる企業またはデータセットのグループ |
| **動的アクセス制御** | アクセス権限は、ユーザーの行動に基づいてセッション中に変化します |
| **セキュリティ ドメイン** | 相互アクセスを防止する分離されたデータ グループ |

Brewer と Nash の **アクセス制御ポリシー** は **ユーザーのアクションに基づいて進化します**。これは、Bell–LaPadula や Biba などの静的モデルにはない独自の特徴です。

#### 仕組み
1. 科目は**制限なし**で始まります。
2. **A 社** のデータにアクセスすると、その対象と **競合企業** (同じ競合クラスの B 社など) のデータとの間に壁が作成されます。
3. 競合関係にない企業へのアクセスは引き続き可能です。

> 📍 これは、**動的フィルタリング** または **メタデータタグ付け** を使用してデータセットを競合クラスにグループ化することで強制されます。

#### 実際の使用例
- **コンサルティング会社**: コンサルタントが競合する 2 つのクライアントと同時に作業することを防ぎます。
- **金融機関**: アナリストが利益相反に基づいて投資決定を下すことを制限します。
- **法律事務所**: 訴訟において顧客間の倫理的な壁を強化します。

#### 視覚的表現
対象者がアクセス → A社データ
→ 紛争クラスが発動しました
→ 同じクラスのB社へのアクセス = 拒否

#### 強みと弱み
| 強み | 弱み |
|------------------------------------|------------------------------------|
| 動的でコンテキストに応じたアクセス | 実装と保守が複雑 |
| 倫理的/法的違反を防止 | 継続的なポリシーの施行が必要 |
| 複数のドメインにわたって柔軟 | 整合性や機密性を直接的に考慮しない |

#### まとめ
- Brewer と Nash モデルは、**利益相反の防止** に重点を置いた **ステートフル**で**コンテキスト認識型のアクセス制御モデル** です。
- 1 つのデータセットにアクセスすると、モデルは**競合/衝突するデータセット**へのアクセスを制限します。
- 従来のアクセス モデルとは異なり、動的であり、以前のアクセス パターンに基づいてリアルタイムで適応します。

⚠️ **CISSP 試験のヒント**: このモデルに関する試験問題では、「利益相反」、「動的アクセス制御」、「倫理的壁」などのキーワードに注意してください。

### モデルにおける「スター」という単語の曖昧さを解消する

**「スター」**という用語は、セキュリティ関連の文献では様々な文脈で登場しますが、使用される場所によって**異なる意味**を持ちます。CISSP受験者にとって、特にアクセス制御モデルやセキュリティフレームワークとの関連において、これらの違いを理解することが重要です。

#### Bell–LaPadula モデルと Biba モデルにおけるスター (*) プロパティ

**Bell–LaPadula** と **Biba** の両方において、**「スター」** (`*` とも表記) という用語は、**多層セキュリティ システムでの書き込み操作を管理する** ルールを指します。

| モデル | スタープロパティ (*) の意味 | セキュリティ目標 |
|--------------------|--------------------------------------------|----------------------|
| Bell–LaPadula | **書き込み禁止**: 被験者はデータを下位の分類レベルに書き込むことができません | 機密性 |
| Biba | **書き込み禁止**: 被験者はより高い整合性レベルに書き込むことができません | 整合性 |

🧠 **重要なルール**:  
- Bell–LaPadula は **機密性** (例: 情報漏洩の防止) に重点を置いています。  
- Biba は **整合性** (たとえば、信頼できないデータが信頼できるデータを破損するのを防ぐ) に重点を置いています。

#### CSA STAR プログラム

**クラウド セキュリティ アライアンス (CSA)** では、**セキュリティ トラスト アシュアランス アンド リスク** プログラムのことを指して **STAR** という頭字語を使用しています。  
これは**セキュリティ モデル**ではなく、**クラウド プロバイダー評価フレームワーク**です。

- CSA STAR = クラウド サービスの透明性と信頼性を向上させるプログラム。
- ベストプラクティスへの準拠を保証するために、サードパーティの監査レポートと自己評価を提供します。

🔐 **CSA STAR レベル**:
- レベル1: 自己評価  
- レベル2: 第三者監査  
- レベル3: 継続的な監視 *(将来の実装用)*

#### ガルブレイスのスターモデル（セキュリティ関連ではありません）

「星」という言葉を使用する別の無関係なモデルは、**ガルブレイスの星モデル**です。  
これは**ビジネス管理フレームワーク**であり、サイバーセキュリティの概念ではありません。

- 企業の戦略、構造、プロセス、人材、報酬を調整するために使用されます。
- CISSP の範囲外です。**セキュリティ モデルと混同しないでください。**

#### 要約表

| 「Star」の使用 | ドメイン | 意味 |
|-------------------|------------------------|-----------------------------------------|
| *-プロパティ (Bell–LaPadula) | アクセス制御モデル | 書き込みを下位の分類に制限します (書き込みなし) |
| *-Property (Biba) | アクセス制御モデル | 書き込みをより高い整合性レベルに制限します (書き込みなし) |
| CSA STAR | クラウド アシュアランス | セキュリティ トラスト アシュアランスおよびリスク プログラム |
| ガルブレイスのスターモデル | ビジネス戦略 | 組織設計モデル（非セキュリティ） |

🛡️ **CISSP 試験のヒント**:  
**Bell–LaPadulaとBibaにおける「*」（星印）は、**書き込み権限に関するものであり、一般的な信頼モデルやクラウド認証に関するものではありません。CSAのSTARやその他のセキュリティ以外のフレームワークと混同しないでください。

## システムのセキュリティ要件に基づいてコントロールを選択する
### コモンクライテリア

**コモンクライテリア（CC）**は、IT製品およびシステムの**セキュリティ機能と保証**を評価および認証するために使用される**国際規格（ISO/IEC 15408）**です。これは、米国の**TCSEC**や欧州の**ITSEC**といった異なる評価規格を**グローバル規格**に統合するために策定されました。

#### 共通基準の目的

- IT 製品の **セキュリティ機能** と **保証レベル** について購入者に **信頼** を提供します。
- **ベンダー**が自社製品の**セキュリティに関する主張**を実証できるようにします。
- 参加国 (米国、英国、フランス、ドイツ、カナダ、日本など) 全体でのセキュリティ評価の**世界的認知**を可能にします。

🛡️ **CISSP 試験のヒント**:  
製品保証を評価する際の**保護プロファイル**、**セキュリティ ターゲット**、および**EAL**の役割を理解します。

#### コモンクライテリアの主要概念

| 用語 | 説明 |
|-----------------------|------------------------------------------------------------------------------|
| **評価対象 (TOE)** | 評価対象となる製品/システム |
| **保護プロファイル (PP)** | 顧客定義のセキュリティ ニーズ - 「必要なもの」 |
| **セキュリティ ターゲット (ST)** | ベンダー定義の実装クレーム - 「提供されるもの」 |
| **パッケージ** | モジュール評価用にグループ化された機能または保証のオプション セット |

- **PP** では、必要な機能と保護について概説します。
- **ST** は、ベンダーの製品が PP 要件をどのように満たしているかを示します。
- PP ↔ ST を一致させることで、購入者は安全な製品を選択できます。

#### 評価保証レベル（EAL）

Common Criteria では、開発とテストの厳しさに応じて **7 つの評価保証レベル (EAL 1 ～ 7)** が定義されています。

| EAL | 名前 | ユースケース/例 |
|------|-----------------------------------|--------------------------------------------------------------------------------------------------|
| 1 | 機能テスト済み | 低リスク環境、基本的な信頼性 (例: 家庭用ファイアウォール) |
| 2 | 構造的にテスト済み | レガシー システム、商用ベスト プラクティス |
| 3 | 体系的にテストおよびチェック済み | 文書化された開発による中程度の保証 |
| 4 | 体系的に設計、テスト、レビュー済み | 最も一般的に使用されており、商用および政府システムで強力な保証 |
| 5 | 半形式的に検証された設計とテスト済み | 形式手法による高度なセキュリティのニーズ |
| 6 | 半形式的に検証およびテスト済み | 非常に高い保証性。重要なインフラストラクチャに使用 |
| 7 | 正式に検証およびテスト済み | 最高の保証; 希少、高価; 機密/国家安全保障システムに使用 |

🔐 **重要**:
- EAL が高いということは、暗号化が強力になったり、脆弱性がゼロになったりするということではない。
- EAL は、セキュリティ機能の**実装に対する信頼性**のみを測定します。

#### コモンクライテリアの強み

- **相互承認協定による国際的な承認**。
- カスタム プロファイルとパッケージによる **柔軟性**。
- 標準化された**透明性の高い評価フレームワーク**。

#### 制限事項

- **実際の使用は評価されません** (例: 管理者の制御、ソーシャル エンジニアリングのリスク)。
- **人的セキュリティ**、**手順管理**、または**物理的保護**を評価しません。
- ベンダーにとって**コストがかかり**、**時間がかかる**可能性があります。

#### まとめ

| 機能 | CC での役割 |
|---------------------|------------------------------------------------------------------------------|
| 評価対象 (TOE) | 評価対象となる製品/システム |
| 保護プロファイル (PP) | 顧客定義のセキュリティ ニーズ |
| セキュリティターゲット (ST) | ベンダーによる PP 要件の実装 |
| 評価保証レベル (EAL) | セキュリティ評価の深さと厳密さを定義します |

🌐 **さらに読む**:  
CC 認定製品とガイダンスの最新情報については、[Common Criteria ポータル](https://www.commoncriteriaportal.org) をご覧ください。

🧠 **CISSP 試験のヒント**:  
**EAL レベル**、**PP と ST**、および Common Criteria 評価の制限 (つまり、カバーされていないもの) を比較する質問が表示される可能性があります。

### 運航許可（ATO）

**運用認可（ATO）**とは、指定された**認可担当者（AO）**が、情報システムが定義された**リスク態勢**の下で運用されることを正式に宣言することです。これは**NISTリスク管理フレームワーク（RMF）**プロセスの重要な成果であり、従来の**「認定」**という用語に代わるものです。

🛡️ **CISSP 試験のヒント**:  
ATO は 1 回限りのアクションではなく、**リスクベース**で期限が定められており、重大な変更やインシデントが発生した後に再検討する必要があることを理解してください。

#### ATOの目的

- システムが**セキュリティ リスク**に関して評価されていることを証明します。
- **残留リスク** が組織にとって許容できるものであることを認識します。
- システムを**ビジネス目的または運用目的**で展開および使用できるようにします。

#### 承認担当者（AO）

- **リスクを受け入れる権限**を持つ上級管理職。
- ATO を発行する前に、システムの評価、脆弱性、および制御を確認します。
- 現在の脅威の状況とシステムの変更に基づいて、承認を変更、取り消し、または更新できます。

#### ATO はいつ必要ですか?

- 新しいシステムや大幅に変更されたシステムを導入する前。
- **規制された環境で運用する場合** (例: 連邦政府機関、国防総省)。
- 重要な後:
  - セキュリティ**侵害またはインシデント**
  - システムの**アップグレードまたはパッチ**
  - **環境または機能**の変化

⏳ **標準的な所要時間**:  
ほとんどの ATO の有効期間は **3 年間** ですが、組織やリスク レベルによって異なる場合があります。

#### 承認決定の種類

| 決定タイプ | 説明 |
|--------------------------------|------------------------------------------------------------------------------|
| **運用承認** | リスクは許容範囲内であり、システムは使用が承認されています |
| **共通コントロール承認** | 継承されたコントロール (共有インフラストラクチャなどから) を承認します |
| **使用承認** | サードパーティのシステム/サービス (クラウド サービスなど) の使用を許可します |
| **承認拒否** | リスクが高すぎるため、システムは動作できません |

📌 **CISSP インサイト**:  
**相互性** が重要です。システムがすでに別の AO で承認されている場合、新しい AO は ATO プロセス全体を繰り返すのではなく、「使用承認」を発行できます。

#### RMF (NIST SP 800-37) との関係

ATO プロセスは、**リスク管理フレームワーク (RMF)** の最終段階の一部です。
1. システムを分類する
2. コントロールを選択
3. 制御を実装する
4. 管理を評価する
5. **承認システム（ATO）**
6. システムを継続的に監視する

#### まとめ

| コンポーネント | 説明 |
|---------------------|------------------------------------------------------------------------------|
| 承認担当者 (AO) | 残留リスクを受け入れる上級担当者 |
| リスクベースの意思決定 | セキュリティ管理と脆弱性を考慮 |
| 有効期間 | 通常 3 年間、または重大な変更/違反が発生するまで |
| RMF の一部 | 制御の評価後に実行され、システムの展開を可能にします |
| 決定タイプ | 承認、拒否、継承された制御の使用またはサードパーティ システムの使用を許可する |

📚 **さらに読む**:  
ATO および RMF プロセスの完全なガイダンスについては、**NIST SP 800-37 改訂 2** を参照してください。  
https://csrc.nist.gov/publications/detail/sp/800-37/rev-2/final

## 情報システムのセキュリティ機能を理解する
### メモリ保護

**メモリ保護**は、プロセスが割り当てられたメモリにのみアクセスすることを保証するため、オペレーティングシステムによって強制される重要なセキュリティ機能です。**システムの安定性**、**データの整合性**、そして**機密性**を維持するための重要な要素です。

🧠 **CISSP 試験の洞察**:  
不適切なメモリ処理は、**バッファ オーバーフロー**、**データ漏洩**、**不正なコード実行** などの脆弱性につながる可能性があります。

#### メモリ保護の目的

- **プロセスを分離**: あるプロセスが別のプロセスのメモリ空間を読み書きするのを防ぎます。
- **OS カーネルを保護します**: ユーザー空間プロセスによるカーネルメモリの改ざんを制限します。
- **マルチタスクをサポート**: 複数のプロセスを相互に干渉することなく同時に実行できます。
- **攻撃を軽減**: **スタックスマッシング**、**ヒープスプレー**、**コードインジェクション**などの悪用手法を防ぐのに役立ちます。

#### 重要なテクニック

| テクニック | 説明 |
|----------------------------|------------------------------------------------------------------------------------------|
| **セグメンテーション** | メモリを論理セグメント (コード、データ、スタックなど) に分割します |
| **ページング** | メモリを固定サイズのブロックに分割し、きめ細かな保護を可能にします |
| **保護リング** | ユーザー モードとカーネル モードの操作を分離します (例: リング 0 とリング 3) |
| **仮想メモリ** | メモリの抽象化を提供し、固有のアドレス空間を介してプロセスを分離します |
| **アクセス制御ビット** | メモリ ページの読み取り、書き込み、実行権限などのフラグ |

🔐 **例**:  
最近のシステムのほとんどは、**ページレベルのアクセス制御** を備えた **ハードウェア強制仮想メモリ** を使用しており、不正なメモリアクセスは **セグメンテーション違反** または **一般保護違反** を引き起こします。

#### メモリ違反とその結果

| 違反の種類 | 結果 |
|----------------------------|------------------------------------------------------------------------------------------|
| **バッファ オーバーフロー** | 隣接するメモリを上書きし、コード インジェクションを可能にします |
| **Null ポインタ参照** | 無効なメモリにアクセスし、クラッシュを引き起こします |
| **競合状態** | 攻撃者が同時操作で共有メモリを操作できるようになります |

#### 実際のエクスプロイトの例

- **CVE-2017-0144 (エターナルブルー)**:  
  Windows上のSMBv1のバッファオーバーフローを悪用。メモリ保護の欠如により任意のコード実行が可能になり、WannaCryランサムウェア攻撃で使用されました。

#### 防衛機制

| 防衛機構 | 説明 |
|----------------------------|------------------------------------------------------------------------------------------|
| **DEP (データ実行防止)** | 実行不可能なメモリ領域でのコード実行を防止します |
| **ASLR (アドレス空間レイアウトのランダム化)** | メモリレイアウトをランダム化して悪用を防止します |
| **スタック カナリア** | オーバーフロー試行を検出するためにスタックに既知の値を挿入します |
| **安全なライブラリ** | メモリセーフな関数を使用します (例: `strcpy()` の代わりに `strncpy()`) |

🚨 **CISSP 警告**:  
メモリ保護を実施しないと、攻撃者が権限昇格やリモート コード実行に悪用できるシステム的な脆弱性が生じます。

#### まとめ
| コンセプト | セキュリティにおける役割 |
|-----------------------|----------------------------------------------------------------------------------|
| メモリ保護 | プロセスの分離とシステムの安定性を確保 |
| セグメンテーションとページング | 境界による構造化メモリ管理を有効にする |
| 仮想メモリ | プロセスレベルの分離を提供 |
| 保護リング | ユーザーとカーネル間の権限分離を強制 |
| 緩和策 | エクスプロイトを防ぐために、DEP、ASLR、スタックカナリア、境界チェックを組み込む |

🛡️ **お持ち帰り**:  
メモリ保護は、オペレーティングシステムとアプリケーションのセキュリティ確保の基盤です。データの破損、システムクラッシュ、コード実行の脆弱性を防ぐ境界を強化します。

### 仮想化
**仮想化**は、ハードウェアを抽象化することで、単一の物理マシン上で複数のオペレーティングシステムやアプリケーションを実行できるようにする、セキュリティ関連技術です。**分離**、**リソース最適化**、そして**テスト**環境において重要な役割を果たします。

🧠 **CISSP 試験の洞察**:  
仮想化は強力なツールであると同時に、潜在的な攻撃対象でもあります。そのメリットとリスクを理解することが、CISSP試験合格の鍵となります。

#### 仮想化の種類
| タイプ | 説明 |
|--------------------------|------------------------------------------------------------------------------------------|
| **サーバー仮想化** | 1 台の物理マシン上で複数の仮想サーバーをホストします |
| **デスクトップ仮想化**| デスクトップ環境を物理デバイスから分離します |
| **アプリケーション仮想化** | 分離されたコンテナーでアプリケーションを実行します |
| **ネットワーク仮想化** | ネットワーク リソースを仮想的に抽象化およびセグメント化します |
| **ストレージ仮想化** | 複数の物理ストレージ デバイスを 1 つの論理ボリュームにプールします |

#### 主要コンポーネント
| コンポーネント | 役割 |
|-------------------|------------------------------------------------------------------------------------------------------|
| **ハイパーバイザー** | 複数の OS がハードウェアを共有できるようにするソフトウェア (タイプ 1 およびタイプ 2) |
| **仮想マシン (VM)** | 別のコンピューターのように動作するエミュレートされた環境 |
| **ゲスト OS** | VM 内で実行されている OS |
| **ホスト OS** | ハイパーバイザーを実行するベース OS (タイプ 2 仮想化) |

#### ハイパーバイザーの種類
| タイプ | 説明 | 例 |
|----------------|----------------------------------------------|--------------------------------|
| **タイプ 1** | ハードウェア (ベアメタル) 上で直接実行 | VMware ESXi、Microsoft Hyper-V |
| **タイプ 2** | 通常のアプリケーションのようにホスト OS 上で実行 | VirtualBox、VMware Workstation |

#### セキュリティ上の利点
- **分離**: VM は論理的に分離されており、1 つの VM が侵害されても他の VM には影響しません。
- **テストとサンドボックス**: マルウェアを分析したり、疑わしいコードを実行したりするための安全な環境。
- **ディザスタリカバリ**: VM スナップショットの簡単なバックアップと復元。
- **リソースの最適化**: ワークロードを統合してハードウェアを効率的に使用します。

🔐 **例**: セキュリティ チームは、侵入テスト ツールを実行したり、隔離された環境で攻撃シナリオをシミュレートしたりするために、VM を使用することが多いです。

#### 仮想化のリスク
| リスク | 説明 |
|----------------------------------|---------------------------------------------------------------------|
| **VM エスケープ** | 攻撃者が VM から抜け出してホスト システムを制御します |
| **ハイパーバイザーの侵害** | ハイパーバイザー レベルの脆弱性により、ホスト全体が侵害される |
| **VM の拡散** | 管理されていない VM が多すぎると、攻撃対象領域が拡大します |
| **スナップショットのリスク** | スナップショットには、攻撃者によって復元される可能性のある機密データが保存される可能性があります|
| **安全でない API** | 仮想化プラットフォームで使用される、保護が不十分な API |

🚨 **CISSP 警告**: ハイパーバイザーの侵害は壊滅的な被害をもたらします。**攻撃対象領域を最小限に抑え、**パッチ適用**と**セグメンテーション**を確実に実施してください。

#### セキュリティ管理
- VM間の通信（東西トラフィック）を制限する
- VM間に厳密なネットワークセグメンテーションを適用する
- ハイパーバイザーには**セキュアブート**とTPMを使用する
- 仮想化管理コンソールを監視する
- ハイパーバイザーへの管理アクセスを制限する

#### まとめ
| コンセプト | 概要 |
|---------------------|------------------------------------------------------------------------------------|
| 仮想化 | 複数の OS 環境におけるハードウェア リソースの抽象化 |
| ハイパーバイザー | ハードウェア アクセスを制御し、VM を分離します |
| セキュリティ上の利点 | サンドボックス、分離、テスト、リソースの最適化 |
| 主なリスク | VM エスケープ、ハイパーバイザー攻撃、VM スプロール |
| コントロール | パッチ適用、監視、アクセス制御、ネットワーク セグメンテーション |

🔐 **お持ち帰り**:  
仮想化は分離を通じて柔軟性とセキュリティを強化しますが、階層化された制御と監視で対処する必要がある新たな脅威をもたらします。

### トラステッド プラットフォーム モジュール (TPM)

**トラステッド プラットフォーム モジュール (TPM)** は、暗号化機能を通じてコンピューティング システムを保護するために設計された専用のハードウェアベースのセキュリティ チップです。主に、**プラットフォームの整合性** の確保、**安全なキーの保存**、および**デバイス ID** の有効化に使用されます。

🛡️ **CISSP 試験の洞察**:  
TPM は **ハードウェアベースの信頼のルート** を提供し、**BitLocker** や **メジャード ブート** などの機能をサポートしていることを理解します。

#### TPM とは何ですか?

- デバイスのマザーボードに組み込まれた **暗号プロセッサ**。
- **Trusted Computing Group (TCG)** 標準によって定義されています。
- 次のような安全な暗号化操作を実行します:
  - キー生成
  - キーとハッシュの安全な保管
  - プラットフォーム構成測定

#### TPM 機能

| 機能 | 説明 |
|-------------------------------|-------------------------------------------------------------------------------|
| **キー ストレージ** | 非対称キーを改ざん防止ハードウェアに保存します |
| **プラットフォーム測定** | ブート プロセス中のシステム状態を記録します (測定ブート) |
| **構成証明** | 暗号証明を使用してシステムが改ざんされていないことを確認します |
| **バインディング** | 特定のシステムでのみ復号化できるようにデータを暗号化します |
| **シーリング** | データを暗号化し、プラットフォームの状態にバインドします。同じ状態でのみアクセス可能です。|

🧠 **バインディングとシーリング**:
- **バインディング** = データをデバイスに結び付ける
- **シーリング** = データをデバイスと構成（例：ブート状態）の両方に結び付ける

#### ユースケース

- **BitLocker ドライブ暗号化**: TPM を使用して暗号化キーを保護します。
- **セキュア ブートとメジャー ブート**: 起動時に OS の整合性を検証します。
- **資格情報ガード**: 機密性の高いログイン資格情報を保護します。
- **デジタル著作権管理 (DRM)** の適用。

#### TPM バージョン

| バージョン | 主な違い |
|---------|----------------------------------------------------------------------|
| 1.2 | RSA + SHA-1; 柔軟性が制限される |
| 2.0 | 最新の暗号化 (ECC、SHA-256 など) をサポート。Windows 11 で使用 |

---

#### TPM と HSM

| 機能 | TPM | HSM |
|----------------|--------------------------------------|--------------------------------------|
| 場所 | デバイス上 (通常はマザーボード) | 外部 (専用アプライアンス/モジュール)|
| 目的 | セキュア ブート、フルディスク暗号化 | エンタープライズ キー管理、暗号操作 |
| パフォーマンス | 中程度 | 高速、高スループット |
| 改ざん防止 | はい | より強力、多くの場合 FIPS 140-2 認定 |

#### セキュリティ上の利点

- ハードウェア レベルで **信頼のルート** を提供します。
- **物理的な攻撃** (例: メモリからのキーの抽出) から保護します。
- **デバイス認証**と安全なシステム検証を有効にします。

🔐 **注**: TPM は、エンドポイント検証が必須である **ゼロ トラスト アーキテクチャ** では特に重要です。

#### セキュリティに関する考慮事項

- TPM は組織によって**有効化され、所有**される必要があります。
- ハードウェア障害に備えて、キーを安全にバックアップする必要があります。
- TPM が侵害されたり故障したりした場合、TPM を交換するには **キーの再生成** または **システムのリセット** が必要になります。
- 脆弱性を修正するために**ファームウェアの更新**を検討してください (特に TPM 2.0 の場合)。

#### まとめ

| トピック | 概要 |
|------------------------|----------------------------------------------------------------------|
| 目的 | ハードウェアベースの安全な暗号化とキーの保存 |
| 一般的な用途 | フルディスク暗号化、セキュアブート、デバイス構成証明 |
| セキュリティ ロール | ハードウェアの信頼のルートとして機能します |
| HSM との比較 | TPM はデバイス上に存在しますが、HSM はネットワークまたは外部に接続されたデバイスです |
| バージョン管理 | 最新の OS (例: Windows 11) には TPM 2.0 が必要です |

🧠 **お持ち帰り**:  
TPMは**プラットフォームセキュリティにとって不可欠**であり、信頼できるブートプロセス、安全な暗号化、そして認証を可能にします。また、**最新のエンドポイント保護**戦略の中核コンポーネントでもあります。

### インターフェース

情報システムにおいて、**インターフェース**は、ユーザーまたはシステムがアプリケーションやコンポーネントとどのようにやり取りするかを定義します。セキュリティの観点から、インターフェースは**制約または制限**され、不正アクセスを防ぎ、ユーザーの**権限またはロール**に基づいて公開される機能を制限する必要があります。

#### 制約されたインターフェースとは何ですか?

**制約されたインターフェース** (**制限されたインターフェース** とも呼ばれます) は、次の基準に基づいてユーザー操作を制限する制御メカニズムです。
- **役割または権限レベル**
- **職務**
- **コンテキストまたは環境**

🛡️ **CISSP 試験のヒント**:  
**Clark–Wilson モデル** は、サブジェクトが **信頼できるプログラムを通じてのみ** オブジェクトにアクセスすることを保証することで、制約付きインターフェースをサポートします。

#### 実装方法

| 方法 | 説明 |
|------------------------|------------------------------------------------------------------------------|
| **メニューの非表示** | ユーザーが利用できないオプションは UI から削除されます。 |
| **コマンドの無効化** | 制限されたオプションは表示されますが、グレー表示されているか無効になっています。 |
| **ロールベースの UI** | インターフェイス全体は、ユーザーの役割またはコンテキストに基づいて動的に調整されます。 |
| **入力フィルタリング** | アクセス レベルに基づいて特定のコマンドまたはパラメータをブロックします。 |

#### ユースケース

- **管理者と通常のユーザー インターフェース**: 管理者はユーザーと設定を管理できますが、通常のユーザーは制限されたアクションしか実行できません。
- **ATM マシン**: インターフェースでは現金の引き出しと残高の表示のみが公開されており、一般ユーザーはバックエンドにアクセスできません。
- **API**: API キー、トークン、またはスコープベースの権限を介して適用され、アクセスできるエンドポイントを制限します。

#### インターフェース制限の利点

- **最小権限の原則** を適用します
- **攻撃対象領域**を縮小
- **偶発的な誤用**や誤った設定を防止します
- アクセス制御モデルの**コンプライアンス**を簡素化

#### シナリオ例

| シナリオ | 制約されたインターフェースの結果 |
|------------------------------------------|------------------------------------------------------------------|
| ゲスト ユーザーがクラウド ダッシュボードにログインします | 請求情報の表示のみ可能で、コンピューティング リソースにアクセスすることはできません |
| ヘルプデスクオペレーターは管理ツールを使用します | パスワードをリセットできますが、アカウントを作成/削除することはできません |
| 開発者はテスト環境の API キーを使用します | ログを読み取ることはできますが、本番環境システムに変更を加えることはできません |

#### 関連するセキュリティモデル

- **Clark–Wilson モデル**: ユーザーは特定のプログラム (変換手順) を通じてのみデータにアクセスする必要があります。
- **RBAC (ロールベースのアクセス制御)**: ロールによって、表示可能またはアクセス可能なインターフェース コンポーネントが決まります。
- **任意アクセス制御および強制アクセス制御** システムでは、権限による制限付きアクセスもサポートされます。

#### まとめ

| コンセプト | 概要 |
|--------------------------|--------------------------------------------------------------------------------------|
| 制約されたインターフェース | ユーザーの権限に基づいて UI またはコマンドを制限します |
| 実装方法 | メニューの非表示、無効化、RBAC ベースのビュー、入力フィルタリング |
| 利点 | エラーを最小限に抑え、最小権限を適用し、攻撃対象領域を削減します |
| サポートされているモデル | Clark–Wilson、RBAC、DAC、MAC |

🔐 **重要なポイント**:  
制約されたインターフェースは、GUI、API、CLI、リモート サービスなど、インタラクション レイヤーで**アクセス制御ポリシー** を適用し、ユーザーが使用を許可されているものにのみアクセスできるようにします。

### フォールトトレランス

**フォールトトレランス**とは、システムのコンポーネントの1つまたは複数に障害が発生した場合でも、システムが**中断することなく**動作を継続する能力です。これはCIAトライアドにおける**可用性**の主要コンポーネントであり、高信頼性システム、ミッションクリティカルな運用、および災害復旧計画に不可欠です。

#### 目的と利点

- **継続的なシステム可用性**を確保する
- **単一障害点**を防ぐ
- **冗長性とフェイルオーバーメカニズム**をサポート
- 障害発生時に**データの整合性と運用の稼働時間**を維持する

🛡️ **CISSP 試験のヒント**:  
**可用性**におけるフォールト トレランスの役割と、それが **ビジネス継続性および災害復旧計画** とどのように統合されるかを理解します。

#### テクニックとテクノロジー

| テクニック | 説明 |
|------------------------|------------------------------------------------------------------------------|
| **RAID (Redundant Array of Independent Disks)** | ディスクに障害が発生してもデータが保持されることを保証します |
| **フェールオーバー クラスタリング** | 障害発生時にワークロードをスタンバイ システムに自動的に移行します |
| **負荷分散** | 過負荷を防ぐために複数のサーバーにワークロードを分散します |
| **冗長電源 / NIC** | 重要なインフラストラクチャ コンポーネントのハードウェア冗長性 |
| **ホット サイト、ウォーム サイト、コールド サイト** | プライマリ サイトに障害が発生した場合に操作を復元するためのリカバリ機能 |

#### RAIDの例

| RAID レベル | フォールト トレランス機能 | 注記 |
|------------|--------------------------------|--------------------------------------|
| RAID 1 | ディスクミラーリング | 冗長性のためにデータが複製されます |
| RAID 5 | パリティ付きストライピング | 1 台のディスク障害にも対応 |
| RAID 6 | ダブルパリティによるストライピング | 2 つのディスク障害にも耐えられます |
| RAID 10 | ミラーリング/ストライピングの組み合わせ | 高パフォーマンスと高可用性 |

#### フェイルオーバーの種類

| タイプ | 説明 |
|---------------------|------------------------------------------------------------------------------|
| **自動フェイルオーバー** | システムが障害を検出すると、すぐに操作をバックアップに移行します |
| **手動フェイルオーバー** | 管理者が介入してバックアップ システムに切り替えます |
| **アクティブ-パッシブ** | バックアップノードは必要になるまでアイドル状態のままです |
| **アクティブ-アクティブ** | すべてのノードがトラフィックを処理します。障害発生時には障害ノードはバイパスされます |

#### 可用性とフォールトトレランス

| 概念 | 定義 |
|-----------------|------------------------------------------------------------------|
| 可用性 | 必要なときにシステムまたはリソースにアクセスして使用する能力 |
| フォールト トレランス | 障害が発生してもシステムが機能し続ける能力 |

#### 設計上の考慮事項

- フォールト トレラント システムは、障害を迅速に **検出** し、**対応** し、**回復** する必要があります。
- 多くの場合、**ヘルス モニタリング エージェント**、**ウォッチドッグ タイマー**、または**冗長コンポーネント** が含まれます。
- 実装にはコストがかかりますが、**継続的な稼働** を必要とするシステム (病院、金融、製造など) には不可欠です。

#### 例

- RAID-6 ディスク、冗長 NIC、デュアル電源、クラスター化されたサーバーを備えた金融取引プラットフォームにより、ストレージ ディスク、ネットワーク カード、またはサーバー全体に障害が発生した場合でも、**取引が継続されます**。

#### まとめ

| コンセプト | 概要 |
|-------------------|------------------------------------------------------------------------------------|
| フォールト トレランス | ハードウェア/ソフトウェア障害時でもシステムが機能し続けることを保証します |
| テクニック | RAID、クラスタリング、負荷分散、電源/ネットワーク冗長性 |
| 目標 | **可用性** と **ビジネス継続性** を維持する |
| 関連ドメイン | 災害復旧、高可用性、システム復元力 |

🧠 **重要なポイント**:  
フォールト トレランスは単なる機能ではなく、システムが操作を中断することなく予期しないコンポーネント障害に耐えられるようにするための **戦略的な設計上の決定** です。

### 暗号化/復号化

**暗号化**と**復号**は、保存時と転送時の両方でデータの**機密性**と**整合性**を保護するための基本的な技術です。暗号化は、アルゴリズムと鍵を用いて平文を暗号文に変換し、復号はその逆のプロセスを実行して元のデータを明らかにします。

#### 重要な概念

| 用語 | 説明 |
|----------------|--------------------------------------------------------------------------------------------|
| **プレーンテキスト** | 暗号化前の読み取り可能なデータ |
| **暗号文** | 暗号化による判読不能なスクランブル出力 |
| **アルゴリズム** | データの暗号化と復号化に使用される数学関数 |
| **キー** | プレーンテキストを暗号化または復号化するためのアルゴリズムで使用される秘密の値 |

🛡️ **CISSP 試験のヒント**: 対称暗号化と非対称暗号化の違いと、それぞれをいつ使用するかを理解します。

#### 対称暗号化

- 暗号化と復号化の両方に**単一の共有キー**を使用します。
- 大容量データでも高速かつ効率的。
- 一般的なアルゴリズム:
  - AES（高度暗号化規格）
  - DES（データ暗号化規格）
  - 3DES（トリプルDES）
  - RC4 (ストリーム暗号、現在は非推奨)

| 機能 | 対称暗号化 |
|------------------|-----------------------------------|
| キー タイプ | 1 つの共有秘密キー |
| スピード | 速い |
| ユースケース | バルクデータ暗号化 (例: ファイル) |
| チャレンジ | 安全なキー配布 |

#### 非対称暗号化

- **キーのペア** を使用します: 1 つは公開キー、もう 1 つは秘密キーです。
- 一方のキーで暗号化したものは、もう一方のキーでのみ復号化できます。
- 使用場所:
  - 安全な通信（例：TLS/SSL）
  - デジタル署名
  - 証明書ベースの認証

| 機能 | 非対称暗号化 |
|------------------|------------------------------------------------|
| キーの種類 | 公開キーと秘密キーのペア |
| 速度 | 対称より遅い |
| ユースケース | 鍵交換、ID 検証 |
| 課題 | 計算オーバーヘッド |

#### 一般的な使用例

| ユースケース | 暗号化の種類 | 説明 |
|--------------------------------|-------------------------|----------------------------------------------------------------|
| ファイル暗号化 | 対称 (AES) | 保存されたデータを効率的に保護 |
| 安全なウェブブラウジング (HTTPS) | ハイブリッド (TLS) | キー交換には非対称、セッションデータには対称を使用 |
| デジタル署名 | 非対称 | 真正性と否認不可性を確認 |
| VPN トンネル暗号化 | 対称 (AES、ChaCha) | パブリック ネットワーク上のプライベート ネットワーク トラフィックを保護 |

---

#### 暗号化関数

| 機能 | 目的 |
|----------------------|------------------------------------------------------------------------------------|
| **機密性** | 許可された関係者だけがデータを読み取れるようにします |
| **整合性** | データが変更されていないことを確認します (多くの場合、ハッシュ + 暗号化経由) |
| **認証** | ユーザーまたはデバイスの ID を確認します (証明書など) |
| **否認防止** | 特にデジタル署名によるアクションの拒否を防止します |

#### 動作モード（ブロック暗号）

| モード | 説明 |
|--------|----------------------------------------------------------------|
| ECB | シンプルだが安全ではない。同じ入力 = 同じ出力ブロック |
| CBC | 初期化ベクトル (IV) を使用して最初のブロックをランダム化します |
| GCM | 認証を追加し、最新の安全なプロトコルで使用されます |

#### CISSPインサイト

- パフォーマンスとユースケースに基づいて、**対称暗号化と非対称暗号化のどちらを選択するべきか**を理解します。
- TLS のような **ハイブリッド システム** がどのように機能するかを理解します (キー交換には非対称暗号化を使用し、セッション データには対称暗号化を使用します)。
- 非推奨のアルゴリズム (DES、RC4 など) とベスト プラクティス (AES-256 の使用など) に注意してください。

#### まとめ

| コンセプト | 概要 |
|------------------------|------------------------------------------------------------------------------------|
| 暗号化 | 機密性を確保するために平文を暗号文に変換します |
| 復号化 | 適切なキーを使用して暗号文を平文に戻します |
| 対称暗号化 | 暗号化/復号化の両方に 1 つのキーを使用します。高速ですが、安全なキー共有が必要です。|
| 非対称暗号化 | 公開鍵/秘密鍵のペア。安全な鍵交換と認証に使用されます |
| 一般的なアルゴリズム | AES、RSA、ECC、3DES、Blowfish、ChaCha20 |

🧠 **重要なポイント**:  
暗号化は**機密保持の要**です。適切な種類とアルゴリズムを選択し、安全な鍵管理を確実に実施してください。

### 情報システムのライフサイクルを管理する

**情報システムライフサイクル（ISLC）**の管理は、安全なITシステムの計画、構築、導入、保守、そして廃止に向けた構造化されたアプローチです。各フェーズにおいて、構想から廃棄に至るまで、セキュリティが確実に組み込まれます。

🧠 **CISSP インサイト**: セキュリティは最後に追加するのではなく、**最初から**組み込む必要があります。

#### 利害関係者のニーズと要件

- **利害関係者が誰であるかを特定します** (ユーザー、管理者、経営幹部、規制機関)。
- 顧客の**ビジネス、セキュリティ、コンプライアンス、パフォーマンスの期待**を理解します。
- これらは**要件収集と検証**の基盤を形成します。

🛡️ **ヒント**: セキュリティ、プライバシー、法的義務が早い段階で確保されるようにしてください。

#### 要件分析

- ステークホルダーのニーズを次のように表現します。
  - **機能要件** (システムが何をするか)
  - **非機能要件** (例: パフォーマンス、セキュリティ、ユーザビリティ)
- 必要な**セキュリティ制御**（例：アクセス制御、暗号化、監査ログ）を特定します。
- 制約事項（予算、規制、既存のシステム）を考慮します。

📌 **CISSP の焦点**: 要件は、**組織のリスク姿勢** と脅威の状況と一致する必要があります。

#### 建築設計

- システムの**設計図**を作成します。
  - システム**コンポーネント**、**データフロー**、**インターフェース**、および**信頼境界**を定義します。
- 適切な**セキュリティ アーキテクチャ モデル** (ゼロ トラスト、多層防御など) を選択します。
- **スケーラビリティ、セキュリティ、コンプライアンス** が設計に組み込まれていることを確認します。

#### 開発/実装

- 設計に基づいてシステムを構築します。
  - **安全なコードを記述**し、ハードウェア/ソフトウェアを構成します。
  - **セキュア SDLC** プラクティスと **セキュア コーディング標準** (例: OWASP) に従います。
- **ユニットテスト**、コードレビュー、バージョン管理を実施します。

🛡️ **重要なリマインダー**: 静的/動的分析と定期的な脆弱性スキャンを統合します。

#### 統合

- さまざまなコンポーネントを 1 つの実用的なシステムに組み合わせます。
- モジュールが**安全に通信**し、設計どおりに機能することを検証します。
- データ フローが **最小権限** とアクセス制御を尊重していることを確認します。

⚠️ 誤って構成された統合は、脆弱性の主な原因です。

#### 検証と妥当性確認

- **検証**: システムを正しく構築しましたか?
- **検証**: 適切なシステムを構築しましたか?
- 含まれるもの:
  - 侵入テスト
  - 機能テスト
  - コンプライアンステスト（例：HIPAA、GDPR）
- セキュリティ テスト ツール (SAST、DAST、IAST、ファジング) を使用します。

🧠 **CISSP のヒント**: 検証 (内部チェック) と検証 (外部有効性) の違いを理解してください。

#### 移行/展開

- システムをステージングから本番環境に移行します。
- 含む：
  - 最終セキュリティ承認
  - デプロイメントスクリプト
  - バックアップとロールバック計画
- ユーザーと管理者をトレーニングします。
- **稼働開始リスク評価**を実施します。

🛠️ CISSP 試験の洞察: 規制された環境に展開する前に ATO (運用許可) が必要になる場合があります。

#### 運用と保守/維持

- システムのセキュリティとパフォーマンスを維持します。
  - パッチを適用する
  - ログとメトリックを監視する
  - アクセスレビューを実行する
- 変化するビジネス、脅威、コンプライアンスのニーズに基づいて更新します。

📌 長期サポートには、**インシデント対応**、**バックアップ/リカバリ**、**構成管理** が含まれる必要があります。

#### 廃止/処分

- システムを安全に廃止する:
  - ストレージメディアを消去または破壊する（DoD/NIST 標準に準拠）
  - **保持ポリシー**に従って必要なデータをアーカイブする
  - すべての資格情報とアクセスを取り消す
- 資産インベントリとドキュメントを更新します。

🚨 不適切な廃棄は**データ漏洩**や**コンプライアンス違反**につながる可能性があります。

#### まとめ

| フェーズ | 主な焦点 |
|----------------------------|---------------------------------------------------------------|
| 利害関係者のニーズ | 期待とビジネス/セキュリティのニーズを理解する |
| 要件分析 | ニーズを機能要件とセキュリティ要件に変換する |
| アーキテクチャ設計 | 多層防御を備えた安全なシステム設計 |
| 開発/実装 | セキュアコーディング、テスト、制御実装 |
| 統合 | コンポーネントを組み合わせて安全な相互運用性を検証 |
| 検証と妥当性確認 | システムがセキュリティと機能の目標を満たしていることを確認する |
| 移行/展開 | バックアップとトレーニング プランを使用して安全に本番稼働を開始 |
| 運用と保守 | システムを安全に監視、パッチ適用、適応 |
| 廃止/廃棄 | 安全なデータ廃棄とシステム廃止の確保 |

🛡️ **CISSP 試験に関する注意**: ISLC は、すべての段階で **リスク管理、セキュリティ テスト、コンプライアンス保証** を統合する必要があります。
